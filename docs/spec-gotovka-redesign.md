# Спецификация: доработка раздела «🥣 Готовка»

Статус: **черновик на утверждение**
Дата: 2026-07-06

## 1. Цель

Раздел «Готовка» должен ощущаться как персональный кулинарный помощник: строгая
навигация по категориям, отсутствие повторов, персонализация по кухням и сезону,
единый визуальный формат карточки с фото, обучение на действиях пользователя —
без повторных опросов.

Затрагиваемые категории: Завтрак, Обед, Ужин, Холодильник (реализуются по общей
системе — очередь, история, персонализация, карточка).

## 2. Текущее состояние (по факту в коде)

- FSM раздела размазана между `store.pending_input`, `onboarding_status`,
  `store.last_action`/`store.last_recipe` (in-memory, не переживает рестарт).
- Кнопка «Ещё рецепт» (`as_food`) всегда генерирует рецепт с constraint
  `"обычное блюдо"`, независимо от того, какую категорию открыл пользователь —
  это и есть баг из п.1 ТЗ.
- Каждое нажатие «Ещё рецепт» / «Заменить» — это новый синхронный вызов LLM
  (`_gen_recipe`/`_gen_leftovers_recipe`, `tier="cheap"`, цепочка Groq → Gemini →
  Claude Haiku → OpenRouter → OpenAI → Cloudflare).
- `m_close` показывает «Готово.» вместо возврата в меню — баг из п.2 ТЗ.
- Анти-повтор есть только для рецептов из холодильника, максимум 12 записей
  (`LEFTOVER_RECIPES_SEEN_KEY`), для обычных рецептов истории нет вообще.
- Кухни пользователя хранятся в `user_settings.json` (`CUISINE_OPTIONS`, 6 кухонь),
  учитываются в промпте через `cuisine_context`, но без веса/приоритета и без
  защиты от повторов одной кухни подряд.
- Сезон, время года, время суток в промпт не передаются.
- Фото рецептов нет — интеграции с Unsplash в проекте нет.
- Кнопка сохранения рецепта (`as_recipe_save`) существует в бэкенде
  (`balance.save_my_recipe` → `my_recipes.json`), но не выведена ни в одной
  клавиатуре карточки рецепта — фактически недоступна пользователю.
- Хранилище — Postgres, единая таблица `kv(key TEXT, value JSONB)`, без ORM и
  миграций; при отсутствии `DATABASE_URL` — in-memory fallback. Все новые сущности
  этой доработки ложатся в эту же модель (новый JSON-ключ per user).

## 3. Решения по открытым вопросам (интервью)

| # | Вопрос | Решение |
|---|---|---|
| 1 | Unsplash-ключ | Через `.env` / переменные окружения: `UNSPLASH_ACCESS_KEY` в `config.py`, читается через `os.environ`, как остальные секреты проекта |
| 2 | Обучение на действиях | Простой счётчик весов по кухням в БД: `+вес` при «Сохранить», `-вес` при «Ещё рецепт» (замена показанного блюда). Веса влияют на вероятность выбора кухни при формировании очереди. Без ML/recency-скоринга |
| 3 | Хранение истории (100 записей) и очереди | В БД проекта (та же `kv`-таблица, новые JSON-ключи per user) — переживает рестарт бота, консистентно с остальными данными |
| 4 | Точность фото Unsplash | Поиск по названию блюда (`name` из ответа LLM) через Unsplash Search Photos API, берём первое релевантное фото |
| 5 | Рецепт из холодильника | Включить в общую систему: `activeMeal=fridge` получает свою очередь, историю, персонализацию по кухне/сезону наравне с завтраком/обедом/ужином — единое поведение для всех 4 категорий |
| 6 | Кнопка «Сохранить» | Восстановить `as_recipe_save` в новых карточках всех 4 категорий — простое использование существующего бэкенда `my_recipes.json` |
| 7 | Формат карточки с фото | `send_photo` с `caption` = текст карточки, одно сообщение. Ограничение Telegram — caption до 1024 символов; при превышении текст карточки сокращается (см. §7) |

## 4. Модель данных (новое)

Все ключи — новые записи в существующей `kv`-таблице (JSONB), per-user, по аналогии
с `config.py` (`FRIDGE_KEY`, `MY_RECIPES_KEY` и т.д.). ORM не используется, паттерн
хранения сохраняется как в остальном проекте.

### 4.1 Активная категория — `ACTIVE_MEAL_KEY = "active_meal.json"`
```json
{"<chat_id>": "breakfast"}
```
Значения: `breakfast | lunch | dinner | fridge`. Устанавливается при явном выборе
категории в меню «Готовка». Сбрасывается (удаляется) при возврате в меню «Готовка»
через «Назад». Никогда не меняется неявно.

### 4.2 Очередь рецептов — `RECIPE_QUEUE_KEY = "recipe_queue.json"`
```json
{
  "<chat_id>": {
    "meal": "breakfast",
    "items": [ {"name": "...", "cuisine": "italian", "...": "..." }, ... ],
    "pos": 0
  }
}
```
- Генерируется целиком (≈10 рецептов) одним вызовом LLM при первом открытии
  категории или когда `pos` достигает конца `items`.
- `meal` в очереди должен совпадать с текущим `active_meal` — если категория
  сменилась, очередь регенерируется, а не переиспользуется.
- «Ещё рецепт» инкрементирует `pos` и отдаёт `items[pos]` без обращения к LLM,
  пока очередь не закончилась.

### 4.3 История рекомендаций — `RECIPE_HISTORY_KEY = "recipe_history.json"`
```json
{"<chat_id>": ["Менемен", "Шакшука", "..."]}
```
- Общая история на пользователя (не разбита по категориям) — блюдо, показанное на
  завтрак, не должно повториться и на ужин.
- Хранит последние 100 названий (FIFO, старые вытесняются).
- Название добавляется в историю в момент генерации очереди (все ≈10 рецептов
  сразу), не в момент показа.
- При генерации новой очереди история передаётся в промпт как список
  «не предлагать»; если LLM не может набрать достаточно уникальных вариантов —
  запрос повторяется без ограничения (допускаются повторы) как самый нижний
  приоритет.

### 4.4 Веса кухонь (обучение) — `CUISINE_WEIGHTS_KEY = "cuisine_weights.json"`
```json
{"<chat_id>": {"italian": 3, "japanese": -1, "turkish": 0}}
```
- Инициализируется нулями/из выбранных в настройках кухонь.
- `+1` при «Сохранить» рецепта данной кухни.
- `-1` при «Ещё рецепт» / «Заменить» для показанного (но не сохранённого) рецепта
  данной кухни.
- Значения используются как модификатор вероятности выбора кухни при формировании
  очереди (см. §5.3), не как жёсткий фильтр — clamp в разумный диапазон, например
  `[-5, +10]`, чтобы одна кухня не «задавила» все остальные.

## 5. Логика генерации очереди

### 5.1 Промпт
Один вызов LLM генерирует список из ~10 рецептов сразу (не по одному). Промпт
формируется из:
- `constraint` (тип приёма пищи: завтрак/обед/ужин, либо список продуктов
  холодильника для `fridge`);
- `priority_context` / `cuisine_context` пользователя (как сейчас);
- весов кухонь (§4.4) — передаются как ранжированный список предпочтений;
- текущего сезона (см. §5.2);
- истории последних рецептов (§4.3) — список «не повторять»;
- требования на разнообразие кухонь внутри самой очереди (см. §5.4).

Формат ответа — JSON-массив из ~10 объектов, каждый с полями:
`name, cuisine, cuisine_emoji, time, servings, ingredients, steps (с длительностью
каждого шага), chef_tip`. Расширяет текущий формат `_gen_recipe` полями
`cuisine`/`cuisine_emoji`/`chef_tip` и структурой шагов с длительностью.

### 5.2 Сезонность
Определяется по текущему месяцу (сервер), без учёта геолокации пользователя:
- Июнь–август → лето: салаты, гриль, лёгкие завтраки, свежие овощи.
- Декабрь–февраль → зима: супы, запеканки, тушёное, более сытные блюда.
- Остальные месяцы → нейтральный промпт без сезонного смещения.
Передаётся в промпт одной строкой-подсказкой, как сейчас передаётся
`priority_context`/`cuisine_context`.

### 5.3 Приоритизация кухонь
При формировании очереди кухни выбираются с вероятностью, пропорциональной
`base_priority (из выбранных в настройках) + вес обучения (§4.4)`, но не жёстко —
это подсказка модели в промпте («предпочтительные кухни в порядке приоритета:
...»), а не программный фильтр по списку кандидатов (нет отдельного этапа
пост-фильтрации сгенерированных названий по кухне).

### 5.4 Анти-однообразие внутри очереди
Промпт явно требует чередования кухонь в サ списке из 10 — не более 2 подряд
рецептов одной кухни, с оговоркой сохранять общий перекос в сторону кухонь с более
высоким весом. Это требование к модели (текстовое правило в промпте), без
дополнительной программной валидации/реролла на этапе разбора ответа (YAGNI —
достаточно правила в промпте с текущим уровнем инструктируемости моделей тира
`cheap`).

## 6. Навигация и FSM

### 6.1 Правило
- Вход в категорию (`a_recipe_breakfast/lunch/dinner`, `as_fridge_cook` первым
  разом) → записать `active_meal`, сгенерировать очередь (если её нет для этой
  категории или `meal` в очереди не совпадает), показать `items[0]`.
- «Ещё рецепт» (`as_food`, единый callback для всех категорий) → прочитать
  `active_meal`, инкрементировать `pos` в очереди этой же категории, показать
  следующий рецепт; при исчерпании — сгенерировать новую очередь для той же
  категории.
- «Назад» (`m_close` в карточке рецепта) → удалить `active_meal` и очередь текущей
  сессии просмотра, вернуть в меню «Готовка» (`menu.send_food_menu`), убрать
  сообщение «Готово.».
- Переход между категориями (Завтрак → Обед и т.п.) возможен только через явный
  выбор из меню «Готовка» — категории никогда не переключаются кнопками внутри
  карточки рецепта.

### 6.2 Изменения в коде (ориентировочно, без готовой реализации)
- `balance.py`: `_recipe_kb()` — единая клавиатура для всех 4 категорий:
  `[✨ Ещё рецепт (as_food)] [❤️ Сохранить (as_recipe_save)] [◀️ Назад (m_close)]`.
  `_recipe_typed_kb()` и `_fridge_recipe_kb()` как отдельные варианты — упразднить,
  свести к общей `_recipe_kb()`.
- `as_food` перестаёт жёстко генерировать `"обычное блюдо"` — читает
  `active_meal` и работает через очередь.
- `m_close` в контексте карточки рецепта — новая ветка поведения (сейчас это
  общий «закрыть», используется и другими разделами — нужно развести по контексту
  вызова, не ломая другие сценарии `m_close`).
- `a_recipe_breakfast/lunch/dinner` и `as_fridge_cook` — точки входа, пишут
  `active_meal` перед первой генерацией очереди.

## 7. Карточка рецепта (UI)

Формат (`ui/food.py`, аналог текущего `food_card`):

```
🍳 Завтрак • 🇯🇵 Японская кухня

Японский омлет

Ингредиенты:
яйца, соль, перец, масло, лук, морковь

Приготовление:
• Взбейте яйца в миске — 2 минуты
• Нагрейте масло на среднем огне — 2 минуты
• Добавьте лук и морковь, обжарьте — 4 минуты
• Залейте яйцами, накройте крышкой — 7 минут

Совет шефа:
<не банальный совет от LLM>

😋 Приятного аппетита!
```

- Эмодзи приёма пищи: 🍳 завтрак, 🥗 обед, 🍽️ ужин, 🧊 холодильник.
- Эмодзи кухни — по флагу страны, отдаётся моделью в поле `cuisine_emoji`
  (с fallback-таблицей на стороне бота на случай пустого/некорректного значения).
- Отправка — `bot.send_photo(chat_id, photo=url, caption=текст_карточки)`, одно
  сообщение. Если `len(caption) > 1024` — сокращаем текст карточки (сначала
  укорачивая описание шагов/совет, сохраняя структуру), а не переходим на два
  сообщения.
- Если Unsplash не вернул фото (ошибка API, нет результатов) — деградация до
  текущего поведения: `send_message` без фото, тот же текст карточки.

## 8. Интеграция с Unsplash

- Новый модуль, например `unsplash.py` (по аналогии с плоской структурой
  проекта — один модуль на внешний сервис, как `ai.py` для LLM).
- `UNSPLASH_ACCESS_KEY` — новая переменная в `config.py`, из `os.environ`,
  без значения по умолчанию (валидация при старте не блокирует бота — фича просто
  деградирует до карточек без фото, если ключ не задан).
- Запрос: Unsplash Search Photos API по `name` рецепта (на английском — модель
  должна возвращать также `name_en` для поиска, либо переводить на лету; решение:
  просить LLM дополнительно вернуть поле `search_query_en` для Unsplash).
- Берётся первая релевантная фотография из ответа. Простое кэширование по
  `name`/`search_query_en` в рамках очереди не требуется (YAGNI) — очередь и так
  генерируется батчем и не должна дёргать Unsplash повторно для одного рецепта в
  рамках одного показа.
- Таймаут и обработка ошибок — как у остальных внешних HTTP-вызовов в проекте
  (синхронный `requests` + `asyncio.to_thread`, по аналогии с `ai.llm_json`).

## 9. Не входит в объём этой доработки (явно вне скоупа)

- Учёт геолокации пользователя для сезонности (только календарная дата сервера).
- ML-модели или recency-скоринг для обучения на действиях — только простой
  счётчик весов.
- Программная пост-валидация ответа LLM на соответствие правилу «не более 2
  кухонь подряд» — полагаемся на промпт-инструкцию.
- Кэш/дедупликация запросов к Unsplash между пользователями.
- Изменение механизма «⭐ Избранное» (`as_fav`/`notes.json`) — остаётся как есть,
  не пересекается с «❤️ Сохранить рецепт».

## 10. Критерий готовности

- Открыть «Завтрак» → «Ещё рецепт» несколько раз подряд → все рецепты только
  завтраки, без повторов внутри истории 100 записей.
- «Назад» из карточки рецепта → попадание в меню «Готовка», без сообщения
  «Готово.».
- Переключение Завтрак → Обед → Завтрак — каждая категория хранит свою очередь и
  позицию независимо.
- Карточка рецепта — с фото (когда Unsplash доступен), с флагом/названием кухни,
  с длительностью по каждому шагу, с советом шефа.
- Кнопка «❤️ Сохранить» кладёт рецепт в `my_recipes.json` и видимо влияет на веса
  кухонь (§4.4) — проверяется через несколько итераций сохранения одной кухни и
  наблюдение возросшей частоты её появления.
- Рецепт из холодильника ведёт себя по той же схеме (очередь, история, карточка,
  сохранение), что и обычные категории.
- Существующие тесты (`tests/test_ui_food.py` и смежные) проходят или обновлены
  под новый формат карточки.

## 11. Порядок реализации (для делегирования сабагентам после утверждения)

Независимые куски:
1. Модель данных + FSM (`active_meal`, очередь, история, веса) в `balance.py`/
   `config.py`/`store.py`.
2. Промпт генерации очереди (расширение `_gen_recipe`/`_gen_leftovers_recipe`
   до батч-генерации 10 шт. с сезоном/весами/анти-повтором/анти-однообразием).
3. Интеграция Unsplash (`unsplash.py` + `config.py`).
4. UI карточки (`ui/food.py`) + единая клавиатура + caption-режим отправки.
5. Навигационные хендлеры (`bot.py`, `balance.py` callback-роутинг) — сведение
   `as_food`/`m_close`/`as_recipe_save` к новой логике.
6. Тесты (обновление `tests/test_ui_food.py`, новые тесты на очередь/историю/веса).

Куски 1–3 можно делать параллельно (не зависят друг от друга по коду, хотя логически
связаны через общий формат данных, зафиксированный в этой спеке). Куски 4–5 зависят
от 1–3. Кусок 6 — после 1–5.
