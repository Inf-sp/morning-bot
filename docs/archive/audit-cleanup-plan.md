# Архив: план чистки и стабилизации архитектуры (PR1–PR5)

Статус: **завершено полностью** · создан 2026-07-06, закрыт 2026-07-08.

Это сжатая архивная сводка исторического плана рефакторинга callback-роутинга
и движка чистки списков. Пользовательское описание того, как раздел «Чистка»
работает сейчас — `docs/cleanup.md`. Здесь — только решения и архитектурные
факты, которые стоит помнить при следующей похожей задаче; пошаговая хроника
реализации и ссылки на удалённые тесты убраны.

**Тесты**: план на всех этапах подтверждался тестами (`tests/test_callback_integrity.py`
и другие) — они упоминались в оригинальной версии документа построчно. Папка
`tests/` с тех пор удалена целиком по решению владельца, автоматических тестов
в проекте больше нет. Ссылки на конкретные тестовые файлы из этой сводки убраны
как неактуальные.

## Зачем всё это делалось

Аудит обнаружил, что статический regex-анализ callback'ов (`verify.audit_callbacks()`)
давал ложноположительный результат: кнопка `set_closet` в пустом гардеробе не
имела обработчика нигде, но проверка этого не ловила, потому что засчитывала
сам факт верхнеуровневого `startswith("set_")` как «обработано», не заглядывая
внутрь под-роутера. Это привело к системной ревизии всей маршрутизации
callback'ов и, параллельно, к обнаружению, что несколько разделов бота
(«чистка» словаря, любимого, гардероба и т.д.) реализовывали одинаковую логику
«список → выбор → удаление» каждый по-своему, без общей защиты от гонки данных.

## Callback integrity — как теперь проверяется маршрутизация

Источник истины — не статический grep по тексту функций, а `routing.py`:
чистая функция `routing.resolve_callback_handler(callback_data) -> dict`,
которая структурно проходит AST исходников `bot.py` и под-роутеров в том же
порядке `if/elif`, что и реальный код, без исполнения handlers (не делает
сетевых вызовов, не трогает LLM/Postgres). Отвечает на вопрос «дошёл бы этот
конкретный `callback_data` до какой-то ветки» — этого достаточно, чтобы ловить
баги вроде `set_closet`, которые regex-аудит пропускал.

`verify.audit_callbacks()` не удалён, но понижен в статусе до вспомогательного
локального линтера — он не источник истины, только быстрая эвристика.

## Правило совместимости для legacy-callback'ов

Ключевое решение, которое стоит помнить при любой будущей миграции маршрута:
**старая inline-кнопка может физически лежать в чатах пользователей месяцами**
(сообщение с клавиатурой продолжает быть нажимаемым, пока Telegram его не
удалит). Отсутствие нажатий в логах за период наблюдения не доказывает, что
кнопкой никто не пользуется — только что её не нажимали в это время.

Поэтому legacy-маршрут никогда не удаляется просто по «тишине в логах» —
вместо этого он становится **чистым редиректом** на канонический маршрут (без
собственного рендера), логирует факт использования для аналитики (не для
решения об удалении) и продолжает работать бессрочно. Так были переведены на
редирект легаси-маршруты стран/артистов/книг (`set_countries`/`set_artists`/
`set_books` → `as_love_countries`/`as_love_artists`/`as_love_books`).

**Важный побочный эффект, вскрывшийся при переводе на редирект**: легаси-путь
`send_countries` читал `COUNTRIES_KEY` (посещённые страны), а канонический
`send_love_section(..., "countries")` читает `FAVCOUNTRIES_KEY` (отмеченные
сердечком любимые). После редиректа старая кнопка «Настройки стран» стала
показывать другой набор данных — это не регрессия, а самостоятельное
исправление более старой путаницы, которое редирект автоматически подтянул
заодно.

**`cfg_*` (легаси-настройки стран/артистов/книг) намеренно не мигрирован** на
современную view-инфраструктуру `cleanup.py` и не удалён — существует
бессрочно как compatibility-слой, пока не будет принято отдельное продуктовое
решение о его судьбе. Это единственный оставшийся в проекте контекст чистки на
старом позиционном формате (без стабильных id и защиты от гонки данных).

## `FAVORITES_KEY` — аудит перед удалением

Путь `send_fav`/`add_fav` (кнопка «В избранное») был недостижим ни одной
кнопкой, но ключ мог накопить исторические данные на проде до более раннего
рефакторинга — поэтому перед удалением потребовался явный SQL-аудит владельцем
бота (все 5 страниц таблицы `kv`, данных не найдено), а не автоматическое
удаление по признаку «код недостижим». После подтверждения пустоты —
константа, функции и callback-путь удалены без миграции.

Тот же принцип позже применён к 8 неиспользуемым storage-ключам
(`CITY_FACTS_KEY`, `CITY_FACTS_DB_KEY`, `TOPICS_NL_KEY`, `TOPICS_EN_KEY`,
`MICRO_TOPICS_KEY`, `MICRO_LESSONS_KEY`, `MICRO_PROGRESS_KEY`, `FOOD_TIP_KEY`) —
удалены одним коммитом только после того, как владелец подтвердил их пустоту
на проде.

## Стабильные id и revision — архитектура `cleanup.py`

Главная миграция: все контексты чистки (кроме `kast_*`, мигрированного раньше
отдельно, и `cfg_*`, см. выше) переведены с позиционной адресации по индексу
элемента на стабильный `id` + `revision` коллекции. Формат элемента —
`ListRecord`: строка оборачивается в `{"id": uuid4_hex, "value": строка}`, а
dict-запись (заметка, страна с флагом) получает только добавленное плоское
поле `"id"` без вложенности — так существующий код, читающий поля записи
напрямую, продолжает работать без переписывания.

Вместо передачи полного UUID в `callback_data` (легко превышает лимит
Telegram 64 байта) введено короткоживущее серверное состояние `view_id` (TTL
24 часа) с коротким суффиксом id в самом callback. Если коллекция изменилась
параллельно с открытием экрана (revision разошлась) или `view_id` устарел —
ничего не удаляется вслепую, показывается «Список уже изменился. Откройте его
заново.», текущий `view_id` инвалидируется.

Миграция проведена частями (не одним большим PR): сначала «Сохранённое»
(самый рискованный список — живёт в чате месяцами), затем Любимое/Скрытое,
затем Словарь/Просмотренное/Прочитанное, затем Холодильник/Рецепты/Здоровье.
На последнем шаге обнаружилось, что `lagom` (принципы пользователя, раздел
Здоровье) хранится не отдельным KV-ключом, а полем внутри профиля
(`memory.get_lagom`/`set_lagom`) — под это `store.py` пришлось обобщить:
`ensure_list_ids`/`remove_from_list_by_ids` стали тонкими обёртками поверх
`ensure_list_ids_via(getter, setter, ...)`/`remove_from_list_by_ids_via(...)`,
принимающих произвольные `getter`/`setter` вместо жёсткого storage-ключа.

## Унификация UI и защита от неявного массового удаления

Финальный этап свёл самодельный чистильщик «битых» слов словаря (кнопка
«🗑 Удалить все» без чекбоксов, без подтверждения) на общий движок —
контекст `d_broken_{lang}`. Заодно введено общее правило: тексты финальной
кнопки называют последствие («Убрать из любимого», «Вернуть в рекомендации»,
«Удалить рецепты» и т.д. — актуальная таблица в `docs/cleanup.md`), и для всех
необратимых коллекций первое нажатие финальной кнопки показывает экран
подтверждения («Удалить N? Это действие нельзя отменить.») — реальное
удаление происходит только после повторного явного нажатия. Обратимые
действия (Любимое, Скрытое) выполняются сразу, без лишнего шага.

## `DIARY_KEY` — принятое решение

Записи вечернего разбора тревог писались в `DIARY_KEY`, но экрана для их
просмотра не существовало. Принято решение (вариант А из нескольких
рассмотренных): ключ остаётся, добавлен полноценный экран «📝 История
самочувствия» в разделе «Здоровье» на той же view-инфраструктуре `cleanup.py`,
без отдельной самодельной реализации. Явно зафиксировано: данные дневника
**не используются** как контекст для LLM-промптов или медицинских
рекомендаций без отдельного будущего продуктового решения и явного согласия
пользователя — добавление экрана просмотра само по себе не разрешает
автоматическое использование истории в остальной логике бота.
