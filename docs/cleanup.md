Каждая находка описана по фиксированной схеме: файл/строка, тип проблемы, что видит
пользователь, риск удаления, решение, тест-подтверждение, приоритет.

Формат нумерации — единый на весь документ, номер = порядок внутри своего уровня
приоритета, не порядковый номер по всему плану:

```text
P3-3 — Неиспользуемые storage keys
P3-4 — Решение по DIARY_KEY
```

Смысл уровней:
- **P0** — видимый пользователю баг (кнопка не отвечает). PR1.
- **P1** — дублирующий живой путь, отсутствующая защита от гонки данных, или

## Раздел A. Callback integrity — усиленная стратегия проверки

Это меняет подход к P0-2 из предыдущей версии плана: статический анализ исходников
(`grep`/AST по `startswith`) остаётся вспомогательным индикатором, но не может быть
единственным источником истины, потому что не воспроизводит реальный порядок
диспетчеризации Telegram → `bot.py` → под-роутер → handler. Основной источник
истины — два уровня динамических тестов.

### Уровень A — сбор реальных callback_data из keyboard factories

Вместо парсинга текста функций (что легко упускает f-строки, условную генерацию
кнопок, `_lv_add_label`-подобные динамические словари), тест **реально вызывает**
каждую публичную функцию, которая строит `InlineKeyboardMarkup`, с синтетическими,
но валидными по форме аргументами (fake `cid`, заранее засеянные тестовые данные
через `store._mem`), и вытаскивает из результата все `callback_data` рекурсивно
из `reply_markup.inline_keyboard`.

Обязательный охват keyboard factories (не исчерпывающий список — тест должен
собирать функции по паттерну имени `_kb`/`_home_kb`/`send_*`, возвращающие
`InlineKeyboardMarkup`, а не полагаться на ручной перечень, чтобы не устаревать):

- `travel._travel_kb()`, `travel._home_kb()`
- кино/книги/музыка: `leisure._movie_kb()`, `leisure._book_kb()`, `leisure._listen_kb()`
- гардероб: `wardrobe.closet_kb()`, `wardrobe._look_result_kb()`, `wardrobe._wardrobe_home_kb()`
- готовка: `balance._recipe_kb()`, `balance._recipe_typed_kb()`, `balance._fridge_recipe_kb()`
- словарь: клавиатуры `learning.py` (дневная подборка, «Проверить словарь», разделы)
- `cleanup.send_cleanup()` — для каждого известного контекста (`d_*`, `nb`, `lv_*`,
  `hid_*`, `cfg_*`, `fridge`, `recipes`, `lagom`, `kast_*`)
- настройки: `settings.send_home()`, `send_love_section()`, `send_wardrobe()`,
  `send_lagom()`, `send_leisure_settings()`, `send_countries/artists/books()`
- onboarding: `firstvisit._skip_kb()`, `_tags_kb()`, `onboard._prio_kb()`
- все остальные публичные функции, возвращающие `InlineKeyboardMarkup`, найденные
  автоматически (не только из списка выше — список выше это минимальный
  обязательный охват, а не потолок)

### Уровень B — dry-run реального роутера (главная проверка)

**Важное уточнение (реализовано именно так): dry-run не исполняет ни один реальный
handler.** Вызывать `bot.answer_callback` целиком с mock-объектами означало бы
реально выполнять бизнес-логику разделов (LLM-запросы, чтение/запись в Postgres,
внешние API) — это дорого, хрупко и не нужно для вопроса «к какому обработчику
уйдёт этот callback». Вместо этого `routing.py` (новый модуль) структурно проходит
**AST** (`ast`-модуль stdlib, без импорта `telegram`/`config`/`store`) исходного
текста `bot.py:answer_callback` и под-роутеров, в том же порядке `if/elif`, в
котором их видит реальный код, и отвечает на вопрос "нашлась ли ветка,
которая приняла бы этот конкретный `callback_data`" — не исполняя её тело.

Публичная функция: `routing.resolve_callback_handler(callback_data: str) -> dict`,
возвращает `{"handled": bool, "module": str|None, "detail": str}`.

Тест обязан **упасть**, если:
- кнопка создаёт `callback_data`, для которого верхнеуровневый роутер `bot.py`
  не находит ни одной подходящей ветки;
- `bot.py` передаёт callback в под-роутер (`settings.handle_callback`,
  `settings.handle_notes_callback`, `wardrobe.handle_callback`,
  `balance.handle_callback`, `myday.handle_callback`, `onboard.handle_callback`),
  но тот не имеет соответствующей ветки (структурно — до конца функции AST не
  находит совпадающего условия);
- `callback_data` превышает лимит Telegram Bot API:

```python
assert 1 <= len(callback_data.encode("utf-8")) <= 64
```

  — это отдельная жёсткая проверка на каждый собранный литерал, не зависящая от
  остальной логики dry-run (лимит documented Telegram API constraint, отдельная
  причина падения теста, чтобы не путать с ошибками маршрутизации).

**Правило allowlist** (реализовано как `_ORPHAN_HANDLER_ALLOWLIST` в
`tests/test_callback_integrity.py`): обратная проверка (handler существует, ни
одна собранная на уровне A кнопка на него не ссылается) не блокирует мердж сама
по себе — но каждый такой случай обязан быть явно перечислен в allowlist с
причиной (compatibility alias / ждёт продуктового решения / часть ещё не
завершённого PR). Тест `test_known_orphan_handlers_are_allowlisted_with_reason`
проверяет, что каждая запись allowlist всё ещё соответствует реальному коду
(handler не удалён без обновления плана) — а `test_all_keyboard_callbacks_resolve_to_handler`
гарантирует, что НИ ОДНА кнопка, реально показанная пользователю, не осталась
без обработчика (это и есть жёсткая, блокирующая часть проверки). Новый
необнаруженный orphan-handler не ломает тесты сам по себе (это не баг видимой
кнопки), но и не появляется в allowlist автоматически — если кто-то хочет
задокументировать временный orphan-путь, запись добавляется вручную вместе с
причиной, так же как это сделано для случаев из PR1/PR2/PR5 ниже.

## Раздел B. Правило совместимости старых callback'ов

Старые inline-кнопки в Telegram могут физически лежать в чатах пользователей
месяцами (сообщение с клавиатурой, отправленное давно, продолжает существовать и
быть нажимаемым, пока Telegram его не удалит/не истечёт). Отсутствие вызовов в
логах за наблюдаемый период **не является доказательством**, что старым callback'ом
никто и никогда не воспользуется — это доказывает только отсутствие нажатий за
период наблюдения, не более.

Правило для всех случаев вида «legacy path + новый path на одну коллекцию»
(P1-2 ниже и любые будущие подобные находки):

- Legacy callback **не имеет собственного рендерера** — только редирект-обработчик,
  который переводит на канонический маршрут:
  ```text
  set_countries → as_love_countries
  set_artists   → as_love_artists
  set_books     → as_love_books
  ```
- Редирект логирует факт использования (`_log.info("legacy callback used: %s", data)`),
  но лог используется только для наблюдения/аналитики, никогда как единственное
  основание для решения об удалении.
- Старые сообщения в чатах продолжают быть рабочими бесконечно долго (redirect —
  не заглушка с «функция переехала», а реальный переход в тот же экран с теми же
  данными).
- Удаление самого alias-обработчика (не рендерера — он уже удалён в момент введения
  alias) — это отдельное продуктовое решение в будущем, не часть автоматической
  чистки этого плана, и не привязано ни к какому таймеру/дедлайну в этом документе.

## PR1 — Callback integrity и видимые баги

### P0-1. Кнопка «✏️ Добавить вещи в шкаф» не отвечает

- **Файл/строка**: `wardrobe.py:214`, `wardrobe.py:662`
- **Тип**: видимый баг — `callback_data="set_closet"` генерируется двумя экранами
  (пустой гардероб в `send_looks`, пустой гардероб в `send_improve`), но обработчика
  `set_closet` нет нигде — ни в `bot.py`, ни в `settings.handle_callback`,
  ни в `wardrobe.handle_callback`.
- **Пользовательский эффект**: пользователь с пустым шкафом нажимает единственную
  предложенную кнопку — бот не реагирует вообще (Telegram покажет «часики» на
  кнопке и тишину).
- **Риск исправления**: нулевой — чистое добавление рабочего пути, ничего не ломает.
- **Решение**: заменить `callback_data="set_closet"` на `callback_data="set_ward_add"`
  в обоих местах — уже существующий рабочий обработчик (`settings.py:566`, ставит
  `pending_input="wardrobe_add_set"` и просит текст с вещами).
- **Тест**: покрывается автоматически `tests/test_callback_integrity.py::test_all_keyboard_callbacks_resolve_to_handler`
  (см. Раздел A); плюс ручная проверка — открыть бота с пустым гардеробом →
  «Образ на сегодня» → нажать «Добавить вещи в шкаф» → бот должен попросить
  описать вещи текстом.
- **Приоритет**: `P0-1`

### P0-2. Callback integrity — статический аудит не ловит реальные баги

- **Файл/строка**: `verify.py:148-181` (`audit_callbacks`), тест
  `tests/test_contracts.py:26-29` (`test_no_unhandled_callbacks`)
- **Тип**: инструмент даёт ложноположительный результат — 0 нарушений, хотя
  `set_closet` явно не обработан.
- **Причина**: `bot.py` роутит `data.startswith(("set_", "setadd_", "setdel_"))` →
  `settings.handle_callback(...)` одной веткой; текущий `audit_callbacks()`
  засчитывает сам факт этого верхнеуровневого `startswith` как «обработано», не
  заглядывая внутрь `settings.handle_callback`, где реальные `elif data == "..."`
  перечислены по одному. Тот же слепой пробел — для `w_*` → `wardrobe.handle_callback`,
  `md_*` → `myday.handle_callback`, `ob_*` → `onboard.handle_callback`, и для веток
  `act = data[2:]` в `bot.py` (`a_<act>`).
- **Пользовательский эффект**: косвенный — баги вроде P0-1 остаются незамеченными
  (`bot.py` вызывает `audit_callbacks()` на старте и логирует warning, не блокируя).
- **Риск**: нулевой, это добавление проверки.
- **Решение**: см. Раздел A целиком — вместо доработки regex-эвристики строится
  двухуровневая динамическая проверка (сбор реальных кнопок + dry-run реального
  роутера). `audit_callbacks()` понижается в статусе до вспомогательного линтера,
  не удаляется.
- **Тест**: `tests/test_callback_integrity.py` (см. Раздел A) — это и есть
  реализация решения, отдельного «теста на тест» не требуется.
- **Приоритет**: `P0-2`

### P1-1. `cleanup.py` — «выбрать всё» (`cla_`) реализовано, но недостижимо ни одной кнопкой

- **Файл/строка**: `cleanup.py:276-286` (`if op == "cla":` в `handle_cleanup`) —
  логика полностью рабочая (toggle всех элементов текущей страницы). Рендер кнопок
  — `cleanup.py:150-202` (`send_cleanup`) — не генерирует `cla_{ctx}_{page}` нигде.
- **Тип**: недостающая кнопка при готовой логике (обработчик есть, кнопки нет —
  решение достроить кнопку, не удалять обработчик).
- **Пользовательский эффект**: массовая очистка коллекции требует отмечать каждый
  элемент вручную даже при явном намерении «удалить всё на видимой странице».
- **Объём PR1 сознательно сужен**: в этом PR реализованы **только** «Выбрать все
  на странице» / «Снять выбор на странице» — две формулировки одной toggle-кнопки
  (`callback_data=f"cla_{ctx}_{page}"`, существующая логика `op == "cla"` без
  изменений в поведении, только текст кнопки меняется в зависимости от текущего
  состояния выбора). Кнопка **«⚠️ Выбрать все N элементов»** (отметить всю
  коллекцию целиком, не только видимую страницу) вынесена в PR3 — она требует
  `view_id`/revision-инфраструктуры (иначе полный выбор коллекции на позиционных
  индексах — тот же риск гонки, что и остальная адресация, см. P1-4) и правил
  подтверждения из PR4, которых в PR1 ещё нет.
- **Решение (реализовано)**: `cleanup.py:send_cleanup` — кнопка появляется, если
  на текущей странице ≥ 2 элементов (`len(chunk) >= 2`); текст —
  «✅ Выбрать все на странице», если хотя бы один элемент страницы не отмечен,
  иначе «✅ Снять выбор на странице»; `callback_data=f"cla_{ctx}_{page}"`.
- **Тест**: `tests/test_cleanup_select_all_page.py` (новый) — кнопка не
  показывается при 1 элементе на странице; появляется при 2+; выбирает/снимает
  именно элементы текущей страницы; явный тест
  `test_select_all_only_affects_current_page_not_whole_collection`, фиксирующий,
  что это **не** «выбрать все N» из PR3.
- **Приоритет**: `P1-1`

## PR3 — Стабильные ID для всех списков (разбито на 4 независимых под-PR)

### Общая модель — вводится один раз в PR3a, переиспользуется в PR3b-d

**`ListRecord`** — общий формат элемента списка:
```text
{"id": <uuid4 hex>, "value": <исходное значение — строка или dict>}
```
Старые элементы получают `id` лениво при первом чтении через миграцию на лету (тот
же паттерн, что уже работает в `store._migrate_legacy_wardrobe`), не единоразовым
batch-скриптом по всей таблице — это снижает риск и не требует downtime.

**Revision коллекции** — счётчик `_v`, инкрементируется при любой мутации списка
(добавление, удаление, массовое действие), по аналогии с уже существующим
`store.mutate_wardrobe`.

**Серверное состояние просмотра `view_id`** — вместо того чтобы пропихивать в
`callback_data` полный `item_id` (32 hex-символа UUID), контекст коллекции,
номер страницы и происхождение (`origin`) одновременно, что легко превышает лимит
Telegram в 64 байта на составных экранах, вводится короткоживущее серверное
состояние:

```text
view_id → {
  collection: str,       # например "notes_fav" или "wardrobe_top_tshirts"
  revision: int,          # снятая на момент открытия экрана версия коллекции
  selected_ids: set[str], # текущий отмеченный выбор (полные uuid хранятся здесь, не в callback)
  page: int,
  origin: str,            # откуда открыт экран, для корректной кнопки "Назад"
  created_at: float,
}
```

`view_id` — короткий случайный идентификатор (например 4-байтовый hex, `secrets.token_hex(4)`),
хранится в том же `store.list_sel`-подобном in-memory/Postgres слое, что и сейчас
хранится выбор, но с TTL (например 24 часа — открытые давно экраны считаются
устаревшими и требуют переоткрытия, что естественно защищает от «протухших»
сообщений в истории чата).

**Формат `callback_data`** — короткий, без полных UUID и версий в открытом виде:
```text
clt:7f9e:ab12    # toggle элемента ab12 (короткий суффикс id, не полный uuid) в рамках view 7f9e
cld:7f9e         # подтвердить действие над всеми выбранными в view 7f9e
clp:7f9e:2       # перейти на страницу 2 в рамках view 7f9e
```
Каждый такой callback проверяется тестом на лимит:
```python
assert 1 <= len(callback_data.encode("utf-8")) <= 64
```

**Обработка устаревшего состояния**: если `view_id` не найден (истёк TTL) или
`revision`, зафиксированная в `view_id`, не совпадает с текущей версией коллекции
на момент нажатия «подтвердить» — ничего не удаляется вслепую, показывается:
```text
Список уже изменился. Откройте его заново.
```
и текущий `view_id` инвалидируется (экран нужно открыть заново, что создаёт новый
`view_id` со свежим `revision`).

### P2-2. Контекстные тексты групповых действий + защита от неявной массовой очистки (выполнено)

Обязательная таблица текстов кнопки финального действия (замена одинаковой
«Удалить отмеченные» на всех экранах):

| Коллекция | Кнопка | Подтверждение |
|---|---|---|
| Любимое | `Убрать из любимого` | Нет (обратимо — можно добавить заново) |
| Скрытое | `Вернуть в рекомендации` | Нет (обратимо) |

| Просмотренное | `Убрать из просмотренного` | Да |

**Правило подтверждения**: необратимые действия (всё, что физически стирает

экрана. Формат:

```text
[🗑 Удалить 47] [◀️ Отмена]
```
«Убрать из любимого» и «Вернуть в рекомендации» подтверждения не требуют — оба
обратимы штатными средствами интерфейса (добавить обратно / нажать повторно).

**Связь с «Выбрать все N элементов»**: кнопка `clx:<view_id>` («🗑 Удалить все N»)
доступна пока только контексту `d_broken_*` (единственный текущий запрос на неё,
см. P2-1) через `_has_select_all_collection_button(ctx)` — но confirm-flow ниже
общий для всех необратимых контекстов независимо от того, как был сформирован
выбор (вручную, «Выбрать все на странице» или «Удалить все N»).

- **Реализация**: `cleanup._action_label(ctx)` возвращает точный текст из
  таблицы выше для каждого контекста (включая `wl`/`rl` → «Убрать из
  просмотренного», `d_broken_*` → «Удалить битые записи»).
  `cleanup._is_reversible_ctx(ctx)` выделяет `lv_*`/`lvls_*`/`hid_*` как
  обратимые — они не проходят через confirm. Для всех остальных `handle_view_
  callback` вводит двухшаговый `cld:`/`cldc:`: первое нажатие финальной кнопки
  (`cld:<view_id>`) переводит `view["confirming"]` в `True` и показывает
  экран подтверждения (`_render_confirm`) вместо удаления; реальное удаление
  происходит только на `cldc:<view_id>` (повторное явное нажатие «🗑 Удалить
  N»). `clcancel:<view_id>` возвращает к обычному экрану чистки без изменений.
  Revision-проверка (защита от гонки, PR3a) действует на обоих шагах — `cld` и
  `cldc` — не только на первом.
- **Тип**: UX-согласованность + защита от случайной потери данных, не просто
  косметика.
- **Проверено вручную**: confirm-экран появляется для необратимых контекстов
  (`d_broken_*`, `fridge`, `recipes`, `lagom`, `nb`, `d_*`, `wl`/`rl`,
  `kast_*`) и не появляется для обратимых (`lv_*`/`hid_*` удаляют сразу, как
  раньше); гонка данных (изменение коллекции между `cld` и `cldc`) блокирует
  удаление на обоих шагах.
- **Приоритет**: `P2-2`

## Решения, принятые владельцем продукта

Зафиксированные ответы на открытые вопросы предыдущей версии плана — не подлежат
повторному пересмотру без явного нового запроса:

1. **`FAVORITES_KEY`** — не мигрируется автоматически в `NOTES_KEY(bucket="fav")`.
   Вместо этого — полный цикл аудита/классификации/выборочного переноса в
   канонические коллекции с отдельным «Импортированное — проверьте» для
   нераспознанного (P1-3).
2. **Legacy callback'и стран/артистов/книг** — не удаляются по признаку тишины в
   логах. Становятся compatibility-редиректами на канонический маршрут, живут
   бессрочно до отдельного будущего решения (Раздел B, P1-2).
3. **`DIARY_KEY`** — вариант А. Ключ остаётся, добавляется экран
   «📝 История самочувствия» в разделе «Здоровье» (P3-4). Данные из дневника не
   используются в LLM-контексте без отдельного будущего решения и явного согласия
   пользователя.
4. **Callback integrity** — основной источник истины — динамический dry-run
   реального роутера (Раздел A), не только статический анализ текста. Существующий
   `verify.audit_callbacks()` не удаляется, но понижается до вспомогательного
   линтера.
5. **Массовые действия** — разделены на «выбрать все на странице» (реализовано в
   PR1, P1-1, без confirm-flow) и «выбрать все N элементов» (переносится в PR3
   вместе с `view_id`/revision — P1-4 — и получает финальное подтверждение по
   правилам P2-2 только там, не раньше).
