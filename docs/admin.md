## 0. Реальность данных (это определяет ВСЁ)

Прежде чем рисовать экраны — что фактически можно посчитать сегодня. Хранилище — простой
KV `kv(key, value JSONB)` в Postgres с откатом в память ([store.py](../store.py)).

| Что есть сейчас | Источник | Годно для |
|---|---|---|
| Список разрешённых пользователей | `access.get_allowed_cids()` | всего пользователей, список |
| Инвайты (pending) | `access.pending_invites()` | инвайты |
| Профиль: имя, город | `store.get_profile()`, `store.get_settings()` | карточка пользователя |
| Онбординг-статус | `onboarding_status.py` | завершил/не завершил |
| Настройки рассылок (вкл/выкл) | `settings.notif_on()` | отключил все рассылки |
| LLM cost-лог (rolling 500) | `ai.get_cost_log()` → `{ts, provider, model, tokens, module}` | расходы, провайдеры, топ-разделы (грубо) |
| Наличие API-ключей | `config.*` | сервисы (конфиг-уровень) |

**Чего НЕТ (потребует нового трекинга):**

| Метрика | Почему сейчас невозможно |
|---|---|
| last_seen / «последний вход» | нигде не пишется время активности пользователя |
| активные сегодня / за 7 дней | нет event-лога действий |
| среднее сообщений на юзера | сообщения не считаются |
| latency LLM, ошибки LLM | cost-лог пишется только при **успехе**, без времени ответа |
| runtime-статус сервисов (🟢/🔴) | нет health-пинга с историей; сейчас только «ключ задан/нет» |
| история ошибок (Логи) | ошибки идут в `logging`/stdout, не персистятся |
| статистика рассылок (отправлено/ошибок/охват) | рассылки не логируют результат |
| популярность разделов (точная) | приблизимо через `module` в cost-логе, но только LLM-разделы |

**Вывод для приоритизации:** экраны делятся на два класса —
**(A) можно сделать сразу** на текущих данных и **(B) сначала нужен слой трекинга**.
Слой трекинга — это 3 маленьких дешёвых примитива (см. §9), после них открывается 80% «взрослой» аналитики.

## 1. Единая система оформления (design system)

Один стиль на все экраны — иначе панель читается как набор разных инструментов.

**Структура любого экрана:**
```
🛠 <Раздел>                    ← заголовок: эмодзи + имя раздела, 1 строка

<1–2 KPI крупно / ключевой статус>   ← «главное за 3 секунды»

<моноширинный блок метрик>     ← выравненные пары «метрика · значение»

<inline-кнопки: действия, потом ◀️ Назад>
```

**Правила (жёстко):**
- **Метрика = одна строка**: `label` слева, значение справа, значение выравнено. Числа — жирным.
- **Максимум 6–8 строк метрик** на экран. Больше — увести на под-экран.
- **Цвет статуса только тремя точками:** 🟢 ок · 🟡 внимание · 🔴 проблема. Никаких других «светофоров».
- **Числа человекочитаемы:** `1.2k`, `18 дней`, `340 мс`, `2 мин назад`. Не сырые timestamp/токены.
- **Дельты, где есть смысл:** `142 (+8 сегодня)`. Тренд важнее абсолюта.
- **Навигация одноуровневая:** любой под-экран → `◀️ Назад` возвращает в раздел, раздел → в Дом. Не глубже 2 уровней.
- **Никакого декора.** Разделители — пустая строка, не «━━━». Эмодзи только как иконка раздела/статуса.

**Шаблон строки метрики** (моноширинный блок, точки-заполнитель для выравнивания на телефоне):
```
Всего ················ 142
Новых сегодня ········ +6
Активны 7д ··········· 38
```

## 2. Боковое меню (Дом админа)

Домашний экран = приборная панель + навигация. Первый экран должен сам отвечать «всё ли ок»,
не заставляя ходить по разделам.

```
🛠 Администратор

Система 🟢 всё работает
Сегодня: 38 активны · 214 LLM · 2 ошибки

[👥 Пользователи]   [📊 Аналитика]
[🤖 LLM]            [📡 Сервисы]
[📢 Рассылки]       [⚠️ Логи]
[⚙️ Настройки]
```

**Решения по структуре меню (переработка исходного):**

| Исходно | Решение | Почему |
|---|---|---|
| LLM + Расходы отдельно | **Объединить: Расходы — под-экран LLM** | Расходы всегда про LLM; в меню 7 пунктов вместо 8, «сначала важное». Кнопка `💰 Расходы` живёт внутри 🤖 LLM. |
| Логи + «Debug/сервисные действия» вперемешку | **Разделить: Логи = только ошибки, сервисные действия → ⚙️ Настройки** | Логи — для чтения проблем, действия (очистка кэша, рестарт) — опасны, им место в Настройках за подтверждением. |
| — | **Добавить строку здоровья на Дом** | Ради цели «10 секунд»: не заходя в разделы видно 🟢/🟡/🔴 и 3 числа дня. |

Итоговое меню — **7 разделов** (как просили), Расходы встроены в LLM.

## 3. 👥 Пользователи

**Экран (A — часть сразу, часть после трекинга):**
```
👥 Пользователи

Всего ················ 142
Новых сегодня ········ +6        ⚠ нужен трекинг
Активны сегодня ······ 38        ⚠ нужен трекинг
Активны 7 дней ······· 71        ⚠ нужен трекинг
Онбординг пройден ···· 118
Не завершили ········· 24
Все рассылки off ····· 9
Сообщений / юзер ····· 12.4      ⚠ нужен трекинг

Последняя активность
👤 Аня · Тбилиси
   Досуг · План поездки
   сегодня 18:41

[🔗 Создать инвайт]
[🔍 Найти пользователя]
[📋 Список] [◀️ Назад]
```

**Формат «последнего входа»** (унифицирован, одна из трёх формулировок по свежести):
- `< 24ч` → `Последний вход: 2 минуты назад` / `сегодня 18:41`
- `1–14 дней` → `Последняя активность: 3 дня назад`
- `> 14 дней` → `Не заходил: 18 дней` (🟡 сигнал оттока)

Эта строка показывается в **трёх местах** (единый компонент `user_activity_line()`):
карточка пользователя, результат поиска, строка в списке.

**Карточка пользователя** (по клику из списка/поиска):
```
👤 Аня · Тбилиси
ID 481… · онбординг ✅

Последний вход ······· 2 мин назад
Активных дней ········ 14 / 30      ⚠ трекинг
Сообщений всего ······ 173          ⚠ трекинг
Любимый раздел ······· 🍿 Досуг     ⚠ трекинг
Рассылки ············· 4 из 6 вкл

[✉️ Написать] [🗑 Удалить]
[◀️ Назад]
```

**Дополнительные метрики активности юзера (предложение):**

| Метрика | Зачем | MVP? |
|---|---|---|
| **Активных дней за 30** | лучший индикатор вовлечённости, чем «был/не был» | позже (нужен трекинг) |
| **Любимый раздел** | понять, за что юзер держится → на чём фокусировать продукт | позже (есть грубо из cost-лога `module`) |
| **Streak (серия дней подряд)** | предсказывает удержание; резкий обрыв = риск оттока | позже |
| **Дата первого входа / «возраст»** | когорты, «старички vs новички» | сразу (дата инвайта) |
| **Статус оттока** 🟢/🟡/🔴 | 🔴 «>14 дней тишины» — кого пора вернуть | сразу (после last_seen) |
| **Часовой пояс / город** | когда слать рассылки, где аудитория | сразу (есть город) |

## 4. 📊 Аналитика

Экран отвечает на «что происходит в продукте», не «что с одним юзером».

```
📊 Аналитика · сегодня

Активны ·············· 38   (7д: 71)
Сообщений ············ 214  ⚠ трекинг
Запросов к LLM ······· 214
Ср. ответ ············ 340 мс ⚠ трекинг
Ошибок ··············· 2
Рассылок отправлено ·· 3

Топ разделов (7д)
🍿 Досуг ············· 41%
📚 Обучение ·········· 22%
👕 Гардероб ·········· 18%
🚑 Здоровье ·········· 11%

[📆 7 дней] [📆 30 дней]
[◀️ Назад]
```

Переключатель периода (сегодня / 7д / 30д) — один и тот же экран, разный срез. «Топ разделов»
считается из `module` cost-лога (сразу, грубо по LLM-разделам; точно — после event-трекинга).

**Дополнительные аналитики (предложение):**

| Идея | Зачем / какую проблему решает | MVP? |
|---|---|---|
| **График активности по часам** (текст-спарклайн `▁▂▄█▅▂`) | когда пик нагрузки → когда слать рассылки, когда греть кэш | позже |
| **Retention D1/D7/D30** | главная метрика здоровья продукта: возвращаются ли | позже (нужны когорты) |
| **Воронка онбординга** (шаг1→шаг2→…) | где отваливаются новички → что чинить в первую очередь | позже |
| **Growth: инвайты → активированы** | эффективность привлечения | сразу |
| **Разделы с ростом/падением** (WoW) | какая фича «взлетает» или «умирает» | позже |
| **Ошибки на 1000 запросов** | качество, а не абсолют ошибок | позже |
| **Пустые/фолбэк-ответы** (доля) | скрытая деградация LLM, которую юзер видит как «тупит» | позже |

- **💰 Расходы** — под-экран (было отдельным пунктом меню, теперь здесь):

```

Запросов ············· 1 214
Токенов (оценка) ····· 486k
Ср. токенов / запрос · 400

Провайдеры (доля токенов)
Gemini ··············· 68%  🟢 ключ
Claude ··············· 24%  🟢 ключ
Groq ················· 8%   🟢 ключ
OpenAI ··············· 0%   ⚪ нет ключа

## 6. 📡 Сервисы

Runtime-здоровье внешних зависимостей. **Сейчас** можем показать «ключ задан + быстрый пинг»;
**полноценно** (latency, последняя ошибка, время проверки) — после health-трекера (§9).

```
📡 Сервисы

🟢 Telegram ·········· 120 мс
🟢 Gemini ············ 340 мс
🟢 Cloudflare ········ 90 мс
🔴 TMDB ·············· 502  ← 12 мин назад
🟡 Ticketmaster ······ медленно 2.1с
🟢 Weather ··········· 210 мс
⚪ Spotify ··········· не настроен

Проверено: 3 мин назад
[🔄 Проверить сейчас]
[◀️ Назад]
```

По каждому: статус · latency · (при ошибке) код + когда. `⚪` — сервис без ключа (не «сломан», а «выключен»).

## 8. ⚠️ Логи

Только то, что сломалось. Читается сверху вниз, свежее — первым.

```
⚠️ Логи · последние ошибки

🔴 22:41 Weather · timeout
🔴 22:03 TMDB · 502
🟡 20:11 Gemini · rate limit
🟢 19:40 (нет ошибок раньше)

Фильтр: [Все] [LLM] [Сервисы] [Рассылки]
[🔎 Подробнее] [🧹 Очистить]
[◀️ Назад]
```

- **Фильтр по источнику** — чипсы.
- **🔎 Подробнее** — полный стектрейс/тело (усечённо, через `secure.redact`, без секретов).
- **🧹 Очистить** — за подтверждением.
- Требует **error-лога** (§9) — сейчас ошибки только в stdout.

## 9. Слой трекинга (backend) — фундамент под (B)

Три дешёвых примитива поверх существующего KV-store. Без них половина метрик невозможна.
Все — rolling-буферы (как cost-лог), чтобы не расти бесконечно.

1. **`activity` — last_seen + счётчик.** На каждый апдейт от юзера: `activity[cid] = {last_ts, count, days:set}`.
   Даёт: активные сегодня/7д/30д, last_seen, сообщений/юзер, streak, статус оттока. **Дешевле некуда** — 1 запись на юзера.
2. **`events` — rolling-лог действий.** `{ts, cid, section, ok}`. Даёт: топ-разделов (точно), нагрузку по часам, воронку, ошибки на 1000. Ограничить ~2–5k записей.
3. **`errors` — rolling-лог ошибок.** `{ts, source, kind, msg}`. Пишется из уже существующего `verify.safe_error` — **точка перехвата уже есть**, нужно только персистить. Даёт: экран Логов, ошибки LLM/сервисов, историю.

+ дополнить **cost-лог полем `ok` и `ms`** (latency) — тогда «ср. ответ» и «ошибок сегодня» станут реальными без нового буфера.

**Порядок внедрения:** `errors` (даёт Логи + ошибки везде, самый дешёвый) → `activity` (оживляет весь раздел Пользователи) → `ms/ok` в cost-лог → `events` (аналитика).

## 10. Ещё 20–30 полезных функций (не декор)

Формат: **зачем · проблема · MVP/позже**.

### Поиск проблем и надёжность
1. **Health-пинг по расписанию + алерт владельцу в ЛС.** Зачем: узнать о падении TMDB раньше юзера. Проблема: сейчас узнаёшь по жалобе. **MVP.**
2. **Circuit-breaker статус на экране Сервисов.** Зачем: видно, что провайдер временно отключён авто-фолбэком. Позже.
3. **«Ошибок всплеск» — авто-баннер на Доме.** Зачем: >N ошибок/5мин → 🔴 на главном. **MVP** (после error-лога).
4. **Кнопка «Тест-прогон всех рассылок в ЛС».** Зачем: проверить контент до массовой отправки. **MVP** (частично есть — превью).
5. **Последний деплой / версия / uptime.** Зачем: «после какого релиза сломалось». **MVP** (дёшево: git-hash + start_ts).
6. **DB-статус + размер ключей.** Зачем: заметить раздувание KV/сбой БД. Позже.

### Пользователи и продукт
7. **Сегменты: активные / спящие / новые / отток.** Зачем: с кем работать. Позже.
8. **«Верни спящих» — рассылка сегменту >14д.** Зачем: реактивация. Позже.
9. **Карта городов (топ-10 текстом).** Зачем: где аудитория → локализация, время рассылок. **MVP** (город есть).
10. **Воронка онбординга по шагам.** Зачем: где теряем новичков. Позже.
11. **Профиль-дифф «до/после онбординга».** Зачем: качество онбординга. Позже.
12. **Экспорт пользователя (GDPR-выгрузка).** Зачем: запрос «покажите мои данные» / поддержка. Позже.
13. **Пометка/заметка на юзере админом.** Зачем: «VIP», «баг у него». Позже.

### Расходы и нагрузка
14. **Бюджет-лимит + алерт «80% за день».** Зачем: не улететь по деньгам. **MVP.**
15. **Стоимость в $ (когда есть usage).** Зачем: реальные деньги, не токены. Позже.
16. **Разбивка расходов по юзерам (топ-транжиры).** Зачем: найти абузеров/тяжёлых. Позже.
17. **Аномалия расхода (сегодня ≫ среднего).** Зачем: цикл/атака/баг «жрёт токены». **MVP** (после ms/ok).
18. **Rate-limit по юзеру + просмотр упёршихся.** Зачем: защита от абуза и счёта. Позже.

### Использование функций
19. **Топ и «мёртвые» разделы.** Зачем: что убрать/раскрутить. **MVP** (грубо из cost-лога).
20. **Фичи без использования N дней.** Зачем: кандидаты на удаление. Позже.
21. **Тренд раздела WoW ↑/↓.** Зачем: реакция на изменения продукта. Позже.
22. **Доля фолбэк/пустых ответов по разделам.** Зачем: где LLM реально «тупит». Позже.

### Операции
23. **Broadcast-таргетинг по сегменту/городу/разделу.** Зачем: релевантные рассылки, меньше отписок. Позже.
