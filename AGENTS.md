# morning-bot

## 1. Роль Claude/Codex

Ты работаешь как инженер продукта, а не как чат.

Цель проекта - Telegram-бот, который помогает одному человеку каждый день: коротко, вовремя и без шума.

Перед любой правкой сначала читай:

1. связанный модуль кода
2. связанный `ui/*.py`
3. связанный файл `docs/*.md`
4. `RELEASE_NOTES.md`
5. стиль соседнего кода

Главное правило продукта:

> Бот не должен выглядеть как набор функций. Он должен ощущаться как личный помощник, который говорит коротко, по делу и вовремя.

---

## 2. Что это за проект

`morning-bot` - персональный Telegram-бот на Python.

Стек:

- Python
- `python-telegram-bot`
- Postgres KV через `store.py`
- fallback в память, если `DATABASE_URL` не задан
- Railway deploy
- UI-сообщения через `MessageBuilder`

Рабочие команды:

```bash
python -m pip install -r requirements.txt
python bot.py
python -m py_compile path/to/changed_file.py
python scripts/sync_version.py
```

Для запуска обязательны `TELEGRAM_TOKEN` и `GEMINI_API_KEY`.
`DATABASE_URL` опционален: без него данные хранятся только до перезапуска.
Railway запускает бот командой `python bot.py` из `railway.json`.

Основные разделы:

- Главное меню
- Мой день
- Погода
- Гардероб
- Обучение
- Досуг
- Поездки
- Готовка
- Здоровье
- Сохранённое
- Настройки
- Ассистент
- Админка

## 4. Главное UI-правило

Во всём боте любая кнопка возврата всегда называется строго:

```text
⬅️ Назад
```

Запрещены варианты:

```text
⬅️ Назад в админ
⬅️ Назад в систему
⬅️ Назад к пользователям
⬅️ Назад к уведомлениям
⬅️ Админ
⬅️ Система
⬅️ Пользователи
```

Это правило выше всех остальных UX-правил.

Если видишь неправильную кнопку возврата - исправь.

Рядом с `⬅️ Назад` всегда стоит вторая кнопка `#️⃣ Меню` (ведёт в главное меню, callback `m_menu`), в одном ряду, два столбца:

```text
⬅️ Назад | #️⃣ Меню
```

Если строка с `⬅️ Назад` уже занята другой кнопкой (например `💾 Сохранить`), `#️⃣ Меню` ставится в паре с `⬅️ Назад` отдельной строкой ниже, а не третьим элементом в ту же строку.

`#️⃣ Меню` всегда отправляет главное меню новым сообщением. Нельзя редактировать
или удалять текущее полезное сообщение: рецепт, рекомендация, учебное задание и
результат должны оставаться в истории чата.

Главное меню, приветствие, онбординг и другие служебные экраны всегда временные:
при следующем сообщении бота они удаляются целиком. Постоянно остаются только
полезные результаты — например рецепты и рекомендации с действием сохранения.
Идентификатор временного экрана хранится в БД, поэтому перезапуск бота не должен
оставлять старое меню в истории. Одновременно в чате может быть только одно меню.

Любая кнопка добавления начинается с `🆕` и называет объект: `🆕 Добавить вещь`,
`🆕 Добавить книгу`, `🆕 Добавить продукт`. В меню она всегда стоит отдельной
первой строкой сразу после описания. Ниже идут остальные действия, последней
строкой — `⬅️ Назад | #️⃣ Меню`.

Любая кнопка, в тексте которой есть «Очистить», начинается с `❌`. Подпись
собирается через `delete_label()` из `ui.constants`, без локального хардкода.

Любое действие или экран выбора называется только в формате `*️⃣ Выбрать …`.
Подпись собирается через `choose_label()` из `ui.constants`, без локального
хардкода эмодзи. Кнопка `*️⃣ Выбрать …` всегда стоит последней содержательной
строкой непосредственно над `⬅️ Назад | #️⃣ Меню`.

## 5. Стиль сообщений

Сообщения должны быть короткими, чистыми и практичными.

Правила:

- после главного заголовка каждого экрана всегда ровно одна пустая строка
- это правило обязательно для старых и новых экранов; новые экраны собирай через `MessageBuilder.title()`
- подписи внутренних блоков (`Что изменено:`, `Ингредиенты:`) идут сразу перед своим содержимым без пустой строки
- один экран - одна главная мысль
- меньше заголовков
- меньше служебных слов
- меньше повторов
- не объяснять очевидное
- не показывать технический мусор
- не писать длинные списки без причины
- важное выделять quote-блоком `>`
- если список нужен - максимум 3-5 пунктов

Хорошо:

```text
🚀 v1.8.7 · Чистые обновления

Что изменено:
- История релизов разбита на короткие пункты
- Служебный шум и повторы удалены

Бот развёрнут и работает ✅
```

Плохо:

```text
🚀 Бот обновлён

Версия: v1.8.7
Статус: успешно запущен

Что изменилось:
...
Деплой прошёл успешно
```

## 6. Стиль кода

Проект пишет короткий Python без лишней архитектуры.

Правила:

- не строить абстракции “на будущее”
- не добавлять слой ради слоя
- не усложнять роутинг
- не дублировать callback-префиксы
- не хардкодить данные, если уже есть источник в `settings.py`, `config.py` или store
- не смешивать UI, store, AI и бизнес-логику в одной функции
- не переписывать рабочий раздел целиком без причины
- сохранять стиль соседнего кода
- если существующий код проще нового варианта - оставить проще

## 8. Правило для `ui/*.py`

Файлы `ui/*.py` - только для рендера.

Можно:

- собирать текст
- собирать кнопки
- использовать `MessageBuilder`
- форматировать карточки
- возвращать готовое сообщение или разметку

Нельзя:

- читать store
- писать в store
- вызывать AI
- делать сетевые запросы
- менять состояние пользователя
- принимать продуктовые решения
- отправлять сообщения напрямую

Если UI-файл начал “думать” - логика лежит не там.

## 9. Роутинг и callback_data

`bot.py` диспетчеризует callback-и по префиксам.

Правила:

- callback-префиксы стабильные
- не плодить похожие префиксы
- не делать callback_data длиннее нужного
- не хранить в callback_data большие данные
- для списков использовать стабильные id
- для опасных действий использовать confirm-экран
- после удаления обновлять текущий экран, а не отправлять новый хаос

`routing.py` используется для проверки реальной маршрутизации callback_data. Он не исполняет handlers.

## 10. Удаление и списки

Для списков и группового удаления используй `cleanup.py`.

Он отвечает за:

- пагинацию
- мультивыбор
- стабильные id
- revision коллекций
- защиту от гонок данных
- confirm-экран перед необратимым действием

Не делай отдельную систему удаления внутри раздела, если задача подходит под `cleanup.py`.

Правильный UX:

```text
Досуг → Кино → Любимое → Убрать из любимого
```

Плохой UX:

```text
Настройки → Удаление → Кино → Любимое → Удалить
```

Пользователь думает об объекте, а не о системном разделе “Удаление”.

## 11. AI Policy: экономия токенов

LLM не используется по умолчанию.

Перед каждым AI-вызовом проверь:

1. Можно ли решить задачу правилами?
2. Можно ли собрать ответ шаблоном?
3. Есть ли готовые данные в store?
4. Есть ли свежий кэш?
5. Нужны ли здесь персональная рекомендация, объяснение или понимание свободного текста?

Если ответ можно получить без LLM - не вызывай LLM.

LLM разрешён для:

- персональных рекомендаций
- объяснения “почему тебе”
- разбора свободной команды пользователя
- языковых объяснений
- суммаризации нескольких источников
- сложных fallback-сценариев

LLM запрещён для:

- меню
- кнопок
- статусов
- пагинации
- удаления
- API-диагностики
- deploy-уведомлений
- простого форматирования
- сортировки
- подсчётов
- выбора из фиксированных списков

## 12. Общий AI-слой

Все AI-вызовы должны проходить через общий слой `ai.py` или аналогичный сервисный модуль.

Разделы бота не должны вызывать LLM напрямую, если уже есть общий AI-слой.
Gemini нельзя вызывать напрямую из разделов `learning`, `leisure`, `food`/`cooking`, `news`, `assistant` и плановых рассылок: только через общий AI-слой.

AI-слой отвечает за:

- кэш
- лимиты
- логирование
- fallback
- выбор модели
- короткий входной контекст
- запрет лишних повторных запросов
- распознавание Gemini 429 / `RESOURCE_EXHAUSTED`
- `last_429_at` и `cooldown_until`

Маршрутизация по типу задачи:

- Cohere первым: обучение, словарь, грамматика, классификация, строгий JSON,
  короткий структурированный анализ в разделах «Здоровье» и «Мысли»;
- Gemini первым: рецепты, гардероб, поездки, досуг, рекомендации, свободный чат
  и другая творческая генерация;
- GitHub Models используется как общий резерв после профильных Cohere/Gemini,
  перед Groq/Cloudflare; модель задаётся через `GITHUB_MODELS_MODEL`;
- исключение — Готовка: `Gemini → Groq → GitHub Models → OpenRouter → кодовый
  fallback из данных Spoonacular`, без Cohere и Cloudflare;
- при ошибке или отсутствии ключа текущего провайдера общий AI-слой пробует
  следующий разрешённый провайдер, не раскрывая ключи в логах.

Каждый AI-вызов должен иметь:

- `feature`
- `user_id`
- `input_hash`
- `cache_ttl`
- короткий prompt
- понятный fallback без AI

Gemini 429 всегда должен иметь cooldown и fallback. Во время активного cooldown новый запрос к Gemini не делается. Дневной лимит не retry-ится; краткосрочный retry допустим только по короткому `Retry-After`.

Плановые уведомления не должны массово генерировать длинный AI-контент. Сначала показывай короткую карточку. Детали генерируй по нажатию пользователя.

## 13. Кэш AI

Дорогие AI-ответы нужно кэшировать.

Персональные данные главных экранов прогреваются фоново один раз в день в 08:00
и при запуске, если актуального кэша ещё нет, затем читаются из кэша при обычном
входе. Принудительная пересборка допустима только по явной кнопке обновления или
после изменения входных данных пользователя.

Ключ кэша должен включать:

```text
user_id
feature
language
level
city
input_hash
profile_version
date_bucket
```

Рекомендуемые TTL:

| Ответ | TTL |
|---|---:|
| Новости | 6-24 часа |
| Рецепт из продуктов | 24 часа |
| Подбор фильма / текущий прокат | 12-24 часа |
| Музыка | 24 часа |
| Живой язык | 7 дней |
| Объяснение грамматики | 30 дней |
| Гардероб по погоде | до смены погоды |
| Deploy message | навсегда для версии |

## 14. Token Budget

У каждого раздела должен быть разумный дневной лимит AI-вызовов на пользователя.

Рекомендуемые лимиты:

| Раздел | AI-вызовы |
|---|---:|
| Мой день | 0-1 в день |
| Погода | 0 |
| Гардероб | 0-1, только сложный подбор |
| Новости | 1 в день |
| Досуг | 1-3 по нажатию |
| Обучение | 1-3 в день |
| Готовка | 1-2 в день |
| Админка | 0 |
| Deploy | 0 |

Главный риск по токенам - плановые уведомления. Они должны быть короткими и ленивыми.

## 15. Сжатый профиль для AI

Не отправляй полный профиль пользователя в каждый prompt.

Используй маленький digest под конкретный раздел.

Пример:

```json
{
  "city": "Alkmaar",
  "country": "Netherlands",
  "languages": {
    "active": "nl",
    "nl_level": "B1",
    "en_level": "A2"
  },
  "style": "modern minimalist, scandi-japandi casual",
  "interests": ["AI", "Apple", "cinema", "music", "Dutch life", "cooking"],
  "avoid": ["long messages", "abstract advice", "technical noise"]
}
```

Правило:

- гардеробу не нужен список любимых фильмов
- новостям не нужен размер обуви
- готовке не нужен DUO
- обучению не нужен список одежды

## 16. Prompt registry

Промты не должны быть разбросаны по коду.

Для новых или выносимых повторно используемых промтов используй целевую структуру:

```text
prompts/
├── news.md
├── leisure_movie.md
├── cooking_from_fridge.md
├── learning_explain_error.md
├── live_language.md
└── assistant_router.md
```

Правила для prompt-файлов:

- короткий prompt
- только нужный контекст
- без повторения глобальных инструкций
- без полного профиля пользователя
- structured output там, где возможно
- Telegram-сообщение собирает `MessageBuilder`, а не LLM

---

## 17. Structured output

Если можно, LLM должен возвращать данные, а не финальный текст Telegram.

Хорошо:

```json
{
  "title": "Чистые обновления",
  "summary": "История релизов стала аккуратнее.",
  "reason": "Теперь меньше повторов и служебного шума."
}
```

Плохо:

```text
Вот красивое сообщение для Telegram...
```

UI собирается отдельно через `MessageBuilder`.

## 18. AI audit log

AI-слой должен логировать:

- `feature`
- `model`
- `tokens_in`
- `tokens_out`
- `cache_hit`
- `cost_estimate`
- `duration_ms`
- `success/error`

В админке показывать коротко:

```text
🧠 AI сегодня

Новости: 1 запрос
Досуг: 2 запроса
Обучение: 3 запроса

Кэш: 68%
Ошибки: 0
```

Не показывать пользователю токены, стоимость и внутренние ошибки.

## 19. Тестирование и проверки

В репозитории есть автоматические тесты, но они не входят в обязательный
рабочий процесс проекта.

Без прямого запроса пользователя:

- не создавать и не обновлять тесты
- не запускать `pytest`
- не использовать тестовое покрытие как критерий готовности
- не писать, что тесты пройдены

Это правило имеет приоритет над `.claude/rules/common/testing.md` и
`.claude/rules/python/testing.md`.

Причина:
Telegram-бот плохо тестируется локально из-за внешних API, Telegram callback-логики, Railway/env-переменных и интеграций. Для текущего проекта важнее точная правка кода и ручная проверка в реальном боте.

Что делать вместо тестов:

1. Внести минимальные изменения в нужные файлы.
2. Кратко описать, что изменено.
3. Дать список ручной проверки в Telegram.
4. Если тесты действительно нужны, сначала спросить пользователя.

Если пользователь этого не просил - тесты не трогать.

## 22. Версии и deploy-уведомления

Версия берётся из `APP_VERSION` или текущей логики проекта.

Не хардкодить версию в шаблоне.

Формат deploy-уведомления админу:

```text
🚀 v1.8.7 · Чистые обновления

Что изменено:
• История релизов разбита на короткие пункты
• Служебный шум и повторы удалены

Бот развёрнут и работает ✅
```

Правила:

- первая строка: `🚀 **vX.Y.Z** · Название` — жирным выделяется только версия
- название короткое, 2-4 слова
- после заголовка всегда одна пустая строка, затем жирная строка `Что изменено:`
- каждое изменение — отдельный пункт `• ...`, одна мысль на пункт
- текст пункта переносится из `RELEASE_NOTES.md` без изменения окончания
- пункты короткие и без quote-блока `>`
- финальная строка: `Бот развёрнут и работает ✅`
- финальная строка обычная, без курсива и жирного выделения
- без блока “Что проверить”
- без технических логов
- уведомление отправляется один раз на новую версию
- обычный рестарт Railway не должен повторять старое уведомление

Fallback, если release notes пустые:

```text
🚀 v{version} · Обновление

Что изменено:
• Бот получил небольшие внутренние улучшения

Бот развёрнут и работает ✅
```

## 24. Мой день

“Мой день” отвечает на вопрос:

> Что мне важно знать сейчас?

Содержит только полезное:

- погода
- предупреждения
- одежда
- короткий совет
- важные события
- напоминания
- полезная рекомендация

Правила:

- заголовок заканчивается городом и маркером `📍`; флаг страны рядом с городом не показывать
- только в «Моём дне» совет называется `🦉Лайфхак:`, без пробела после эмодзи
- лайфхак содержит одно конкретное действие, условие или способ выполнения и понятный практический результат
- общие наблюдения, мотивационные фразы, банальности и непроверяемые обещания не считать лайфхаком
- не превращать бриф в газету
- не повторять одно и то же в разных блоках
- давать действие, а не просто факт
- для ADHD-формата писать коротко и конкретно
- погода ведёт к решению: зонт, куртка, обувь, велосипед, UV
- если данных нет - сказать коротко

Плохо:

```text
Сегодня возможны осадки, влажность высокая, ветер юго-западный.
```

Хорошо:

```text
Дождь после обеда. Возьми дождевик, особенно если поедешь на велосипеде.
```

## 25. Погода

Погода должна быть практичной.

Фокус:

- дождь
- ветер
- ощущается как
- UV
- риск для велосипеда
- что надеть
- брать ли зонт или дождевик

Правила:

- не показывать лишнюю метео-таблицу
- не писать “погода недоступна” без причины, если можно дать fallback
- учитывать лимиты OpenWeather
- кэшировать, когда это предусмотрено логикой
- не тратить API-запросы без необходимости
- AI не нужен

## 26. Гардероб

Гардероб помогает быстро выбрать образ по погоде и стилю.

Категории вещей:

```text
Верх
Низ
Верхняя одежда
Обувь
Аксессуары
```

Правила:

- не использовать эмодзи в названиях категорий
- “Верх”, не “👕 Верх”
- “Низ”, не “👖 Низ”
- не писать блок “Чуть точнее образы будут...”, если он не нужен
- учитывать погоду, дождь, ветер, температуру
- учитывать стиль пользователя
- показывать готовый образ, а не случайный список вещей
- при удалении вещи обновлять образ или список
- управление вещами держать внутри гардероба
- базовый подбор делать правилами, без LLM
- физические свойства хранить отдельно от названия вещи; тепло — только `лёгкие`, `обычные` или `тёплые`
- в жаркую погоду полностью исключать `тёплые` вещи до оценки цвета и стиля
- в карточке сохранённой вещи не показывать исправление: только `❌ Удалить` с подтверждением
- в карточке образа после погодной строки показывать только `Надень:` и один финальный блок `💡 Полезно:`
- после `Надень:` не оставлять пустую строку; каждую вещь выводить с новой строки с маркером `•`
- `💡 Полезно:` содержит конкретное действие, которое улучшает посадку, способ носки или практичность образа, и объясняет результат
- блоки `Как носить`, `Почему работает` и `Образ готов` в карточке образа не показывать
- фразы `ничего добавлять не нужно`, `без стилистических изменений`, `образ готов` и другие пустые подтверждения запрещены
- в «Моём стиле» показывать только активные переключатели; цвета — `тёмные`, `светлые`, `яркие`, причём `яркие` не являются запретом
- в жару и солнце отдавать приоритет солнцезащитным очкам, но только если они есть в реальном шкафу
- в шкафу использовать публичные категории `Футболки и рубашки`, `Брюки и шорты`, `Куртки и пальто`, `Обувь`, `Аксессуары`; поиск не показывать
- `🆕 Добавить вещь`, `👕 Мой шкаф` и `*️⃣ Выбрать стили` размещать отдельными широкими строками на своих экранах

Главный смысл:

> Пользователь не должен думать у шкафа.

## 27. Обучение

Обучение - единая система для языков.

Активный язык обучения - одна общая настройка для всего бота.

Она влияет на:

- тренажёр слов
- тренажёр фраз
- живой язык
- обучающие уведомления
- словарь
- игровые механики

Поддерживаемые языки:

- нидерландский
- английский

Правила:

- активный язык один
- уровень хранится отдельно для каждого языка
- экран настроек обучения показывает активный язык и уровни
- во вкладке “Обучение” показывается только активный язык
- прогресс показывается прямо на главном экране обучения; отдельной кнопки `📊 Прогресс` нет
- языки не смешиваются в одном тренажёре
- после смены уровня можно предложить стартовые слова нового уровня

## 28. Тренажёр слов

Цель - быстро учить нужные слова.

Правила:

- карточка короткая
- слово, перевод, короткий пример
- после добавления писать, что именно добавлено
- указывать язык и категорию, если полезно
- не добавлять дубли
- использовать пользовательский словарь
- стартовые слова брать по `language` и `level`
- первый запуск ограничивать разумным числом слов
- страницы делать короткими

Хорошо:

```text
Добавил в словарь:

> innemen — принимать лекарство

Язык: нидерландский
```

---

## 29. Тренажёр фраз

Тренажёр фраз должен быть чистым.

Запрещено писать:

```text
Выбери подходящее слово:
```

Это понятно по кнопкам.

Запрещено повторять одно и то же предложение в задании и объяснении без причины.

Правильная структура:

```text
🧩 Проверь себя

U moet deze tabletten twee keer per dag ____
```

После ошибки:

```text
Не совсем.

Правильно:
> U moet deze tabletten twee keer per dag innemen.

`innemen` здесь значит “принимать лекарство”.
```

Правила:

- короткая фраза
- один фокус
- понятная ошибка
- кнопка разбора правила
- кнопка повторить
- кнопка следующая фраза
- не писать “смотри значение в переводе фразы”

## 30. Живой язык

“Живой язык” учит реальным фразам прямо в главной карточке обучения.

Формат:

```text
Живой язык: Laat maar. → Ладно, забудь / забей
Когда говорят? Когда больше не хотят объяснять, спорить или продолжать тему.
```

Правила:

- без академических объяснений
- живой перевод
- одна короткая ситуация употребления
- один вариант на весь календарный день
- без AI и сетевого ожидания
- отдельной кнопки, карточки и действия «Ещё вариант» нет
- уведомление «Слова и фразы дня» решает только задачу повторения ранее изученного материала

## 31. Игра-детектив

Игра должна учить через интерес.

Правила:

- один эпизод - одна маленькая задача
- без длинного квеста
- без перегруза текстом
- язык соответствует активному языку обучения
- уровень соответствует уровню пользователя
- подсказка помогает, но не решает всё
- финал закрепляет фразу или слово

## 32. Словари и языки

Раздел отвечает за:

- активный язык обучения
- уровень нидерландского
- уровень английского
- пользовательский словарь
- добавленные слова
- знакомые слова

Правила:

- язык обучения не прячется в общих настройках
- лишние языки не показываются во вкладке обучения
- активный язык указан явно
- смена языка влияет на все обучающие функции
- при ручном добавлении слова бот пишет, в каком виде оно сохранено
- слово или фраза сохраняются в словарь сразу, без ожидания подтверждения кнопкой; для нидерландской записи первой отдельной строкой показывать `🔊 Прослушать`, затем `❌ Удалить`, `📋 Мой словарь` и нижнюю строку `⬅️ Назад` / `#️⃣ Меню`; кнопку `✅ Готово` не показывать
- одна кнопка озвучивания произносит только нидерландский термин целиком, через паузу 700 мс — нидерландский пример; перевод, формы и разбор не озвучивать
- Azure Speech использовать с голосом из env (по умолчанию `nl-NL-MaartenNeural`) и скоростью `-10%`; результат отправлять через `sendVoice`, а Telegram `file_id` сохранять в устойчивом кэше по языку, голосу, скорости, термину и примеру
- callback озвучивания хранит только стабильный `wordId`; повторное нажатие не обращается к Azure, а ошибка TTS не меняет и не удаляет карточку
- явная команда добавления в словарь имеет приоритет над любым открытым pending-вводом, включая `😮‍💨 Мысли`; варианты вроде `Добавить *twijfelt*` должны добавлять чистый термин `twijfelt` и сбрасывать прежний режим ожидания
- слова, фразы и другие типы записей используют общий шаблон карточки; нидерландский глагол — специальный компактный вариант с формами, типом и `💡 Пример:`
- термин в карточке словаря выделяется только жирным и начинается с большой буквы
- перевод всегда показывается в той же строке через обычную стрелку `→`, без отдельной подписи «Перевод: …»
- заголовок статуса карточки («✅ Добавлено»/«✅ Обновлено»/«📚 Найдено») — отдельной строкой, без термина на той же строке
- блока «Примеры» и кнопки «Другой перевод» в карточке нет
- обычная карточка заканчивается единственным блоком `💡 Полезно:` без курсива; карточка нидерландского глагола использует `💡 Пример:`
- в `💡 Полезно:` показывается ровно один короткий, естественный и частотный разговорный пример; редкие, книжные и сложные конструкции запрещены
- пример и его перевод разделяются только стрелкой `→`
- нидерландский глагол определять по AI-части речи, не только по окончанию `-en`; после этого получать строгий JSON через Cohere с Gemini и GitHub Models fallback
- все самостоятельно написанные нидерландские ответы тренажёра проверять через LanguageTool после локального сравнения с эталоном и альтернативами; стилистическое замечание не делает ответ неправильным
- простые исправления показывать без LLM; спорную грамматику и смысл объяснять через `Groq → Gemini`, максимум одним предложением
- перед сохранением нидерландской записи отдельно проверять термин, примеры и формы, никогда не отправлять в LanguageTool русский перевод и не блокировать сохранение при недоступности сервиса
- «Обновить базу» проверяет нидерландские и английские учебные записи пакетами, с ограничением параллельности, retry и резервной копией; безопасные исправления применяются автоматически, смысловые изменения требуют решения пользователя
- не использовать отдельную команду проверки текста; карточки замечаний всегда форматирует код без rule id, сырого JSON и технических сообщений
- для глагола показывать `инфинитив · imperfectum singular · perfectum`, тип `weak/strong/irregular` переводить как `слабый/сильный/неправильный глагол`
- глагольные формы, пример и уверенность валидировать до показа; ниже `0.75` либо при неполных формах показывать `Формы: не удалось проверить`
- ошибочный анализ не блокирует добавление: сохранить базовую запись и показать `Не удалось получить формы глагола.` без технической ошибки
- валидный анализ хранить в самой записи словаря с `analysis_provider=app_llm`; повторно использовать сохранённые поля и не создавать общую базу глаголов

- события

- рекомендации объясняют “почему тебе”
- не подбирать случайное
- учитывать любимое и сохранённое

- показывать коротко
- давать быстрые кнопки действия
- настройки досуга держать внутри досуга
- не показывать кнопки и отдельные экраны `Скрытое`, `Скрытые артисты`, `Смотрел`, `Прочитано` и `Уже знаю` в категориях Кино, Книги, Музыка, Страны и Любимое
- скрытые, просмотренные/прочитанные/знакомые и удалённые из рекомендательных списков элементы записывать в единую внутреннюю категорию `Не рекомендовать`, независимо от исходного раздела
- единый стоп-лист хранит тип объекта только для технической фильтрации; не показывать отдельные стоп-листы внутри Кино, Книг, Музыки или Поездок
- LLM использовать только для персонального вкуса и объяснения
- книжные метаданные и обложку сначала проверять через Google Books; выбирать
  совпадение по названию и автору, Open Library использовать только как резерв
- отсутствие `GOOGLE_BOOKS_API_KEY` или внешний сбой не должны блокировать
  карточку книги; редакторский текст LLM внешними метаданными не перезаписывать

## 38. Готовка

Готовка помогает решить, что приготовить.

Разделы:

- завтрак
- обед
- ужин
- из того, что есть
- сохранённые рецепты
- настройки

Правила:

- не повторять рецепты подряд
- основной рецепт получать из Spoonacular: продукты пользователя → 10 кандидатов →
  Gemini выбирает лучший, переводит и адаптирует, а не генерирует блюдо с нуля
- для Готовки соблюдать порядок `Gemini → Groq → GitHub Models (openai/gpt-4.1-mini) → OpenRouter`;
  если все LLM недоступны, собрать карточку кодом из Spoonacular
- формат карточки всегда создаёт код; LLM возвращает только структурированные данные
- сохранять технические `spoonacular_id` и `spoonacular_source_name`; TheMealDB с
  публичным `THEMEALDB_API_KEY=1` использовать как резерв
- винное сочетание брать только из `pairing_wines` Spoonacular; безалкогольное
  сочетание может добавлять Gemini при адаптации; сбой внешнего API не блокирует Готовку
- учитывать продукты пользователя
- разделять завтрак, обед и ужин
- не показывать общее время приготовления
- писать обычно 3 компактных шага без отдельного времени; простому блюду можно 2, сложному максимум 4
- объединять связанные действия; каждый шаг начинать с глагола и ограничивать 1–2 короткими предложениями
- не делать длинные кулинарные статьи
- любой блок с подписью «Полезно:» оформлять только как `💡 Полезно:`
- не показывать фотографию блюда и не тратить AI/vision-запросы на фото
- винное и безалкогольное сочетания объединять без эмодзи в одну строку `К блюду подойдет: …`; не показывать отдельные подписи «Сочетания» и «Без алкоголя»
- в «Моём холодильнике» использовать только шесть категорий в фиксированном порядке: `Мясо и рыба`, `Овощи и фрукты`, `Молочное и напитки`, `Бакалея`, `Специи и соусы`, `Заморозка`
- на кнопке категории показывать только название и количество доступных продуктов через ` · `; не использовать эмодзи, дроби и категорию «Прочее»
- все замороженные продукты относить в `Заморозку`; при низкой уверенности в категории нового продукта просить пользователя выбрать одну из шести категорий
- кнопку `❌ Удалить продукты` показывать только на главном экране холодильника непосредственно над нижней навигацией
- внутри категории использовать `🟢` для доступного продукта и `🔴` для закончившегося

## 39. Здоровье

Здоровье — справочные сценарии и внешняя память, не врач и не психотерапия.

Разделы:

- врачебная справка
- `😮‍💨 Мысли`
- короткие действия
- лагом
- простые действия

Правила:

- не ставить диагноз
- не делать медицинские выводы без оснований
- медицинский текст направлять во «Врача», кризисный — в прямой протокол 112/113
- для общего разбора давать от одного до трёх коротких выводов и ровно одно конкретное следующее действие
- не писать философию
- не давить мотивацией
- каждое сообщение сохранять одной мыслью; убирать только лишние пробелы, переносы и начальный маркер, не меняя смысл
- классификацию и служебные признаки не показывать
- активные записи показывать прямо на главном экране без счётчика; единственную кнопку `🧐 Разобрать мысли` скрывать при пустом списке
- передавать в общий разбор полный активный список, его количество, локальные дату и время пользователя
- ожидать JSON с `summary` до 20 слов, `analysis` из 1–3 пунктов до 20 слов и одним `next_step`
- разделять тревогу, уже принятое решение и реальное действие; не утверждать, что тревога необоснованна
- не советовать `Записать мысль`, `Сменить обстановку`, `Вернуться к текущему делу`, `Выбрать действие на 5–15 минут` или `Отложить мысль до отдельного времени`
- невалидный или шаблонный JSON заменять fallback-разбором со ссылками на конкретные мысли пользователя
- под общим разбором показывать вертикально только `🕒 Оставить на потом` и `❌ Удалить мысли`
- `🕒 Оставить на потом` сохраняет записи и результат, закрывает разбор и возвращает главный экран с теми же мыслями
- `❌ Удалить мысли` только после подтверждения физически удаляет записи текущего разбора; не затрагивает записи вне разбора, настройки и персонализацию
- после закрытия хранить лишь обезличенный факт завершения разбора; массовое удаление всей истории мыслей не показывать в пользовательском интерфейсе
- не показывать статистику тревожности, фильтры и теги
- `ZEROENTROPY_API_KEY` использовать только для поиска по закрытой проверенной базе

Лагом-тон:

```text
Сейчас не вся жизнь. Сейчас один шаг.
```

## 44. Админка

Админка должна быть компактной.

Правила:

- меньше кнопок верхнего уровня
- диагностика внутри “Система”
- API, LLM и новости можно объединять в одну системную диагностику
- тесты уведомлений держать во вкладке “Уведомления”
- кнопка “Проверить всё” не дублируется в разных местах
- все кнопки возврата только `⬅️ Назад`
- ошибки показывать одной строкой
- технические подробности прятать глубже
- лимиты API показывать просто, не сырой статистикой
- системные сервисы показывать как `эмодзи · сервис · раздел · статус или статистика`
- у зелёного сервиса не писать «работает», «доступен», «активен» или «в норме»
- `🔴` ставить только при реально неработающей функции без резерва; рабочий резерв означает `🟡`
- HTTP-коды, traceback и имена исключений показывать только во вложенном экране `⚠️ Логи`
- журнал ошибки хранит раздел, действие, полный текст, traceback, файл, строку,
  функцию, сервис, резерв и версию; любые секреты обязательно редактируются

- осталось, если важно

- повторы
- длинные raw errors
- stack traces
- HTTP-коды и имена исключений
- лишние нули

## 49. Внешние API

- fallback должен быть простым и честным
- внешние ошибки не показывать пользователю как raw error

## 51. Рабочий алгоритм

Перед правкой:

1. Найди связанный модуль.
2. Найди связанный `ui/*.py`.
3. Найди docs-файл раздела.
4. Проверь `RELEASE_NOTES.md`.
5. Посмотри стиль соседнего кода.

Во время правки:

1. Меняй минимально нужный участок.
2. Не ломай callback-и.
3. Не меняй UX соседних разделов без запроса.
4. Не добавляй тесты.
5. Не добавляй архитектуру без причины.
6. Следи за кнопкой `⬅️ Назад`.
7. Не добавляй прямой LLM-вызов в раздел, если есть AI-слой.

После правки:

1. Обнови docs.
2. Обнови `RELEASE_NOTES.md`.
3. Проверь сценарий вручную или через маленький `python3 -c`.
4. Кратко напиши, что изменено.
5. Честно напиши, что проверено.
6. Не обещай то, что не проверял.
