# morning-bot

Telegram-бот на Python (`python-telegram-bot`), персональный ассистент на каждый день:
погода и утренний бриф, гардероб, обучение языкам, досуг (кино/книги/музыка/поездки),
готовка, здоровье, заметки и настройки. Хранилище — Postgres KV (`store.py`) с откатом
в память, если `DATABASE_URL` не задан.

## Структура кода

Модуль на раздел бота: `wardrobe.py`, `myday.py`, `leisure.py`, `travel.py`,
`learning.py`, `balance.py`, `settings.py`, `menu.py`, `admin.py`, `assistant.py`,
`onboard.py`/`onboarding_status.py`/`firstvisit.py` — плюс общий роутер `bot.py`,
диспетчеризующий `callback_data` по префиксам. `ui/*.py` — чистые функции рендера
сообщений (`MessageBuilder`), без побочных эффектов и обращений к store/AI.

Инфраструктурный слой: `store.py` (Postgres KV с откатом в память) + `config.py`
(константы, ключи хранилища, чтение переменных окружения) + `routing.py`
(AST-резолвер реальной маршрутизации `callback_data`, используется для проверки
целостности колбэков, не исполняет handlers) + `access.py`/`secure.py` (доступ и
безопасность) + `memory.py`/`tracking.py`/`verify.py`/`skills.py` (вспомогательные
сквозные механизмы). `cleanup.py` — общий движок пагинации, мультивыбора и
защищённого от гонки данных группового удаления (стабильные id + revision
коллекций + confirm-экран перед необратимым действием) для списков всех разделов,
описан в `docs/cleanup.md`.

## Тестирование

В проекте нет автоматических тестов (`pytest`, `tests/`) — папка тестов была
удалена целиком по решению владельца, и новые тесты сознательно не пишутся.
Не предлагай запускать pytest, не создавай файлы `test_*.py` и не описывай
тестовое покрытие как критерий готовности задачи. Корректность проверяется
чтением кода, ручным прогоном сценария (например через `python3 -c` с
синтетическими данными) и, для UI/frontend-подобных изменений, реальным
прохождением сценария в боте.

## Документация разделов — `docs/*.md`

Каждый раздел бота (Гардероб, Мой день, Досуг → Кино/Книги/Музыка/Путешествия,
Готовка, Обучение, Здоровье, Настройки, Главное меню) имеет файл в `docs/` с
фиксированной структурой:

```text
# Название раздела
## Назначение        — зачем раздел существует, одна-две мысли
## Как это работает   — сценарий от входа в раздел до результата
## Основные возможности — список конкретных функций
## Настройки          — что можно настроить и где
## Ограничения        — честные технические/продуктовые границы
## Что можно улучшить — известные пробелы
## Обновления         — журнал изменений (см. ниже)
```

Раздел «Как это работает» и «Основные возможности» должны явно описывать
**навигацию**: что пользователь видит сразу при входе в раздел (главный экран/панель
состояния), какие подкатегории/вложенные меню там есть (например, в кино — список
жанров и раздел «по настроению»; в гардеробе — иерархия Зона → Подкатегория), и
**связи с другими разделами** (например: приоритеты из «Настроек» влияют на подбор
образа в «Гардеробе»; язык обучения выбирается в «Обучении», а не в общих настройках).
Не описывай реализацию (имена функций/файлов) — только то, что видит и может сделать
пользователь.

Файлы docs называются короткими английскими slug-ами, но заголовок внутри файла
совпадает с пользовательским названием раздела или кнопки в UI. Текущая карта:

### Обязательное правило: обновлять docs вместе с кодом

**Каждый раз**, когда меняется поведение раздела (новая функция, изменённый флоу,
другая структура меню, убранная возможность) — соответствующий файл в `docs/`
обновляется в той же задаче, не откладывая на потом. Это не отдельный шаг «по
запросу», а часть определения готовности задачи.

- Если правишь код одного раздела — открой и поправь его файл в `docs/`.
- Если меняешь общее меню/навигацию (`menu.py`, `bot.py` роутинг) — поправь
  `docs/menu.md`.
- В конце файла есть `## Обновления` — добавляй туда короткую запись
  (дата, что изменилось) при значимых изменениях, не при любой мелкой правке.
- Если создаёшь новый раздел — заведи новый файл `docs/<slug>.md` по той же структуре.

`docs/instructions.md` — отдельная инструкция для администратора бота (не
пользовательский раздел, поэтому вне карты и структуры выше): расписание
плановых рассылок, структура админ-панели, обязательные/опциональные
переменные окружения, диагностика проблем. Обновляй её вместе с кодом, если
меняется расписание `run_daily`/`run_repeating` в `bot.py`, состав экранов
`admin.py`, набор читаемых переменных окружения в `config.py` или доступные
команды `CommandHandler`.

## ECC-правила

Локальный поднабор конвенций лежит в `.claude/rules/` (слой `common` + `python`);
подробности и приоритет — в `.claude/rules/README.md`. Стиль проекта терсный
(короткие функции, минимум абстракций) и местами сознательно расходится с
дефолтами ECC — при конфликте ориентируйся на существующий код, а не на общий шаблон.
