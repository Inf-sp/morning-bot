# morning-bot

## 1. Роль Claude/Codex

Ты работаешь как инженер продукта, а не как чат.

Цель проекта - Telegram-бот, который помогает одному человеку каждый день: коротко, вовремя и без шума.

Перед любой правкой сначала читай:

1. `CLAUDE.md`
2. связанный модуль кода
3. связанный `ui/*.py`
4. связанный файл `docs/*.md`
5. `RELEASE_NOTES.md`

Главное правило продукта:

> Бот не должен выглядеть как набор функций. Он должен ощущаться как личный помощник, который говорит коротко, по делу и вовремя.

---

## 2. Что это за проект

`morning-bot` - персональный Telegram-бот на Python.

Стек:

- Python
- `python-telegram-bot`
- Postgres KV через `store.py`
- fallback в память, если `DATABASE_URL` не задан
- Railway deploy
- UI-сообщения через `MessageBuilder`

Основные разделы:

- Поездки

## 4. Главное UI-правило

Во всём боте любая кнопка возврата всегда называется строго:

```text
⬅️ Назад
```

Запрещены варианты:

```text
⬅️ Назад в админ
⬅️ Назад в систему
⬅️ Назад к пользователям
⬅️ Назад к уведомлениям
⬅️ Админ
⬅️ Система
⬅️ Пользователи
```

Это правило выше всех остальных UX-правил.

Если видишь неправильную кнопку возврата - исправь.

Рядом с `⬅️ Назад` всегда стоит вторая кнопка `#️⃣ Меню` (ведёт в главное меню, callback `m_menu`), в одном ряду, два столбца:

```text
⬅️ Назад | #️⃣ Меню
```

Если строка с `⬅️ Назад` уже занята другой кнопкой (например `⭐️ Сохранить`), `#️⃣ Меню` ставится в паре с `⬅️ Назад` отдельной строкой ниже, а не третьим элементом в ту же строку.

`#️⃣ Меню` всегда отправляет главное меню новым сообщением. Нельзя редактировать
или удалять текущее полезное сообщение: рецепт, рекомендация, учебное задание и
результат должны оставаться в истории чата.

Любая кнопка добавления начинается с `🆕` и называет объект: `🆕 Добавить вещь`,
`🆕 Добавить книгу`, `🆕 Добавить продукт`. В меню она всегда стоит отдельной
первой строкой сразу после описания. Ниже идут остальные действия, последней
строкой — `⬅️ Назад | #️⃣ Меню`.

## 5. Стиль сообщений

Сообщения должны быть короткими, чистыми и практичными.

Правила:

- после главного заголовка каждого экрана всегда ровно одна пустая строка
- это правило обязательно для старых и новых экранов; новые экраны собирай через `MessageBuilder.title()`
- подписи внутренних блоков (`Что изменено:`, `Ингредиенты:`) идут сразу перед своим содержимым без пустой строки
- один экран - одна главная мысль
- меньше заголовков
- меньше служебных слов
- меньше повторов
- не объяснять очевидное
- не показывать технический мусор
- не писать длинные списки без причины
- важное выделять quote-блоком `>`
- если список нужен - максимум 3-5 пунктов

Хорошо:

```text
🚀 v1.8.7 · Чистые обновления

Что изменено:
- История релизов разбита на короткие пункты
- Служебный шум и повторы удалены

Бот развёрнут и работает ✅
```

Плохо:

```text
🚀 Бот обновлён

Версия: v1.8.7
Статус: успешно запущен

Что изменилось:
...
Деплой прошёл успешно
```

## 6. Стиль кода

Проект пишет короткий Python без лишней архитектуры.

Правила:

- не строить абстракции “на будущее”
- не добавлять слой ради слоя
- не усложнять роутинг
- не дублировать callback-префиксы
- не хардкодить данные, если уже есть источник в `settings.py`, `config.py` или store
- не смешивать UI, store, AI и бизнес-логику в одной функции
- не переписывать рабочий раздел целиком без причины
- сохранять стиль соседнего кода
- если существующий код проще нового варианта - оставить проще

## 8. Правило для `ui/*.py`

Файлы `ui/*.py` - только для рендера.

Можно:

- собирать текст
- собирать кнопки
- использовать `MessageBuilder`
- форматировать карточки
- возвращать готовое сообщение или разметку

Нельзя:

- читать store
- писать в store
- вызывать AI
- делать сетевые запросы
- менять состояние пользователя
- принимать продуктовые решения
- отправлять сообщения напрямую

Если UI-файл начал “думать” - логика лежит не там.

## 9. Роутинг и callback_data

`bot.py` диспетчеризует callback-и по префиксам.

Правила:

- callback-префиксы стабильные
- не плодить похожие префиксы
- не делать callback_data длиннее нужного
- не хранить в callback_data большие данные
- для списков использовать стабильные id
- для опасных действий использовать confirm-экран
- после удаления обновлять текущий экран, а не отправлять новый хаос

`routing.py` используется для проверки реальной маршрутизации callback_data. Он не исполняет handlers.

## 10. Удаление и списки

Для списков и группового удаления используй `cleanup.py`.

Он отвечает за:

- пагинацию
- мультивыбор
- стабильные id
- revision коллекций
- защиту от гонок данных
- confirm-экран перед необратимым действием

Не делай отдельную систему удаления внутри раздела, если задача подходит под `cleanup.py`.

Правильный UX:

```text
Досуг → Кино → Любимое → Убрать из любимого
```

Плохой UX:

```text
Настройки → Удаление → Кино → Любимое → Удалить
```

Пользователь думает об объекте, а не о системном разделе “Удаление”.

## 11. AI Policy: экономия токенов

LLM не используется по умолчанию.

Перед каждым AI-вызовом проверь:

1. Можно ли решить задачу правилами?
2. Можно ли собрать ответ шаблоном?
3. Есть ли готовые данные в store?
4. Есть ли свежий кэш?
5. Нужны ли здесь персональная рекомендация, объяснение или понимание свободного текста?

Если ответ можно получить без LLM - не вызывай LLM.

LLM разрешён для:

- персональных рекомендаций
- объяснения “почему тебе”
- разбора свободной команды пользователя
- языковых объяснений
- суммаризации нескольких источников
- сложных fallback-сценариев

LLM запрещён для:

- меню
- кнопок
- статусов
- пагинации
- удаления
- API-диагностики
- deploy-уведомлений
- простого форматирования
- сортировки
- подсчётов
- выбора из фиксированных списков

## 12. Общий AI-слой

Все AI-вызовы должны проходить через общий слой `ai.py` или аналогичный сервисный модуль.

Разделы бота не должны вызывать LLM напрямую, если уже есть общий AI-слой.
Gemini нельзя вызывать напрямую из разделов `learning`, `leisure`, `food`/`cooking`, `news`, `assistant` и плановых рассылок: только через общий AI-слой.

AI-слой отвечает за:

- кэш
- лимиты
- логирование
- fallback
- выбор модели
- короткий входной контекст
- запрет лишних повторных запросов
- распознавание Gemini 429 / `RESOURCE_EXHAUSTED`
- `last_429_at` и `cooldown_until`

Каждый AI-вызов должен иметь:

- `feature`
- `user_id`
- `input_hash`
- `cache_ttl`
- короткий prompt
- понятный fallback без AI

Gemini 429 всегда должен иметь cooldown и fallback. Во время активного cooldown новый запрос к Gemini не делается. Дневной лимит не retry-ится; краткосрочный retry допустим только по короткому `Retry-After`.

Плановые уведомления не должны массово генерировать длинный AI-контент. Сначала показывай короткую карточку. Детали генерируй по нажатию пользователя.

## 13. Кэш AI

Дорогие AI-ответы нужно кэшировать.

Персональные данные главных экранов прогреваются фоново один раз в день в 08:00
и при запуске, если актуального кэша ещё нет, затем читаются из кэша при обычном
входе. Принудительная пересборка допустима только по явной кнопке обновления или
после изменения входных данных пользователя.

Ключ кэша должен включать:

```text
user_id
feature
language
level
city
input_hash
profile_version
date_bucket
```

Рекомендуемые TTL:

| Ответ | TTL |
|---|---:|
| Новости | 6-24 часа |
| Рецепт из продуктов | 24 часа |
| Подбор фильма / текущий прокат | 12-24 часа |
| Музыка | 24 часа |
| Живой язык | 7 дней |
| Объяснение грамматики | 30 дней |
| Гардероб по погоде | до смены погоды |
| Deploy message | навсегда для версии |

## 15. Сжатый профиль для AI

Не отправляй полный профиль пользователя в каждый prompt.

Используй маленький digest под конкретный раздел.

Пример:

```json
{
  "city": "Alkmaar",
  "country": "Netherlands",
  "languages": {
    "active": "nl",
    "nl_level": "B1",
    "en_level": "A2"
  },
  "style": "modern minimalist, scandi-japandi casual",
  "interests": ["AI", "Apple", "cinema", "music", "Dutch life", "cooking"],
  "avoid": ["long messages", "abstract advice", "technical noise"]
}
```

Правило:

- гардеробу не нужен список любимых фильмов
- новостям не нужен размер обуви
- готовке не нужен DUO
- обучению не нужен список одежды

## 19. Тестирование и проверки

В проекте нет автоматических тестов.

Папка `tests/` была удалена по решению владельца. Новые тесты не писать.

Нельзя:

- предлагать `pytest`
- создавать `test_*.py`
- писать “тесты пройдены”, если тестов нет
- описывать тестовое покрытие как критерий готовности
- добавлять тесты “на всякий случай”

## Правило по тестам

Не создавать, не обновлять и не запускать тесты без прямого запроса пользователя.

По умолчанию:

* не добавлять файлы в папку `tests/`;
* не менять существующие тесты;
* не запускать `pytest`;
* не писать отчёты вида `python3 -m pytest -q`;
* не тратить время и лимиты на тестирование.

Причина:
Telegram-бот плохо тестируется локально из-за внешних API, Telegram callback-логики, Railway/env-переменных и интеграций. Для текущего проекта важнее точная правка кода и ручная проверка в реальном боте.

Что делать вместо тестов:

1. Внести минимальные изменения в нужные файлы.
2. Кратко описать, что изменено.
3. Дать список ручной проверки в Telegram.
4. Если тесты действительно нужны, сначала спросить пользователя.

Если пользователь этого не просил - тесты не трогать.

- если меняешь расписание или переменные окружения - обнови `docs/instructions.md`
- если создаёшь новый раздел - создай новый docs-файл
- не описывай внутренние функции в пользовательской документации
- описывай только то, что видит и может сделать пользователь

`docs/instructions.md` - отдельная инструкция администратора. Она не является пользовательским разделом.

---

## 21. Release notes

После каждых правок обновляй `RELEASE_NOTES.md`.

Правила:

- запись обязательна даже для маленькой правки
- писать продуктовым языком
- не писать имена функций без необходимости
- не писать технические подробности без необходимости
- не повторять одно и то же разными словами
- 1-4 коротких пункта достаточно
- каждый пункт начинается с `- ` и содержит только одно изменение
- не объединять несколько действий или результатов в один длинный абзац
- точки в конце пунктов не ставить
- если началась другая категория работ - поднять patch-версию

Плохо:

```text
Исправлена функция send_deploy_notification, добавлен guard от duplicate railway restart.
```

Хорошо:

```text
- Бот сообщает о новой версии один раз
- Обычный рестарт не повторяет старый отчёт
```

## 22. Версии и deploy-уведомления

Версия берётся из `APP_VERSION` или текущей логики проекта.

Не хардкодить версию в шаблоне.

Формат deploy-уведомления админу:

```text
🚀 v1.8.7 · Чистые обновления

Что изменено:
- История релизов разбита на короткие пункты
- Служебный шум и повторы удалены

Бот развёрнут и работает ✅
```

Правила:

- первая строка: `🚀 **vX.Y.Z** · Название` — жирным выделяется только версия
- название короткое, 2-4 слова
- после заголовка всегда одна пустая строка, затем жирная строка `Что изменено:`
- каждое изменение — отдельный пункт `- ...`, одна мысль на пункт
- пункты короткие, без точки в конце и без quote-блока `>`
- финальная строка: `Бот развёрнут и работает ✅`
- финальная строка обычная, без курсива и жирного выделения
- без блока “Что проверить”
- без технических логов
- уведомление отправляется один раз на новую версию
- обычный рестарт Railway не должен повторять старое уведомление

Fallback, если release notes пустые:

```text
🚀 v{version} · Обновление

Что изменено:
- Бот получил небольшие внутренние улучшения

Бот развёрнут и работает ✅
```

# Правила разделов

## 23. Главное меню

Главное меню - входная точка в бот.

Оно должно:

- быстро показывать основные разделы
- не перегружать пользователя
- не дублировать настройки каждого раздела
- вести в понятные подменю
- сохранять одинаковую логику названий

Правила:

- названия кнопок короткие
- пользователь понимает, куда нажимает
- пользовательские разделы и админка не смешиваются
- новая кнопка не добавляется, если её смысл уже есть внутри раздела
- при изменении структуры меню обновляется `docs/menu.md`

---

## 24. Мой день

“Мой день” отвечает на вопрос:

> Что мне важно знать сейчас?

Содержит только полезное:

- погода
- предупреждения
- одежда
- короткий совет
- важные события
- напоминания
- полезная рекомендация

Правила:

- не превращать бриф в газету
- не повторять одно и то же в разных блоках
- давать действие, а не просто факт
- для ADHD-формата писать коротко и конкретно
- погода ведёт к решению: зонт, куртка, обувь, велосипед, UV
- если данных нет - сказать коротко

Плохо:

```text
Сегодня возможны осадки, влажность высокая, ветер юго-западный.
```

Хорошо:

```text
Дождь после обеда. Возьми дождевик, особенно если поедешь на велосипеде.
```

---

## 25. Погода

Погода должна быть практичной.

Фокус:

- дождь
- ветер
- ощущается как
- UV
- риск для велосипеда
- что надеть
- брать ли зонт или дождевик

Правила:

- не показывать лишнюю метео-таблицу
- не писать “погода недоступна” без причины, если можно дать fallback
- учитывать лимиты OpenWeather
- кэшировать, когда это предусмотрено логикой
- не тратить API-запросы без необходимости
- AI не нужен

---

## 26. Гардероб

Гардероб помогает быстро выбрать образ по погоде и стилю.

Категории вещей:

```text
Верх
Низ
Верхняя одежда
Обувь
Аксессуары
```

Правила:

- не использовать эмодзи в названиях категорий
- “Верх”, не “👕 Верх”
- “Низ”, не “👖 Низ”
- не писать блок “Чуть точнее образы будут...”, если он не нужен
- учитывать погоду, дождь, ветер, температуру
- учитывать стиль пользователя
- показывать готовый образ, а не случайный список вещей
- при удалении вещи обновлять образ или список
- управление вещами держать внутри гардероба
- базовый подбор делать правилами, без LLM
- физические свойства хранить отдельно от названия вещи; тепло — только `лёгкие`, `обычные` или `тёплые`
- в жаркую погоду полностью исключать `тёплые` вещи до оценки цвета и стиля
- в карточке сохранённой вещи не показывать исправление: только `❌ Удалить` с подтверждением
- «Почему работает» писать одним живым предложением без повторов слова `комплект` и механических оборотов
- в «Моём стиле» показывать только активные переключатели; цвета — `тёмные`, `светлые`, `яркие`, причём `яркие` не являются запретом
- в жару и солнце отдавать приоритет солнцезащитным очкам, но только если они есть в реальном шкафу
- в шкафу использовать публичные категории `Футболки и рубашки`, `Брюки и шорты`, `Куртки и пальто`, `Обувь`, `Аксессуары`; поиск не показывать
- `🆕 Добавить вещь`, `👕 Мой шкаф` и `Выбрать стили` размещать отдельными широкими строками на своих экранах

Главный смысл:

> Пользователь не должен думать у шкафа.

---

## 27. Обучение

Обучение - единая система для языков.

Активный язык обучения - одна общая настройка для всего бота.

Она влияет на:

- тренажёр слов
- тренажёр фраз
- живой язык
- обучающие уведомления
- словарь
- игровые механики

Поддерживаемые языки:

- нидерландский
- английский

Правила:

- активный язык один
- уровень хранится отдельно для каждого языка
- экран настроек обучения показывает активный язык и уровни
- во вкладке “Обучение” показывается только активный язык
- прогресс показывается прямо на главном экране обучения; отдельной кнопки `📊 Прогресс` нет
- языки не смешиваются в одном тренажёре
- после смены уровня можно предложить стартовые слова нового уровня

---

## 28. Тренажёр слов

Цель - быстро учить нужные слова.

Правила:

- карточка короткая
- слово, перевод, короткий пример
- после добавления писать, что именно добавлено
- указывать язык и категорию, если полезно
- не добавлять дубли
- использовать пользовательский словарь
- стартовые слова брать по `language` и `level`
- первый запуск ограничивать разумным числом слов
- страницы делать короткими

Хорошо:

```text
Добавил в словарь:

> innemen — принимать лекарство

Язык: нидерландский
```

---

## 29. Тренажёр фраз

Тренажёр фраз должен быть чистым.

Запрещено писать:

```text
Выбери подходящее слово:
```

Это понятно по кнопкам.

Запрещено повторять одно и то же предложение в задании и объяснении без причины.

Правильная структура:

```text
🧩 Проверь себя

U moet deze tabletten twee keer per dag ____
```

После ошибки:

```text
Не совсем.

Правильно:
> U moet deze tabletten twee keer per dag innemen.

`innemen` здесь значит “принимать лекарство”.
```

Правила:

- короткая фраза
- один фокус
- понятная ошибка
- кнопка разбора правила
- кнопка повторить
- кнопка следующая фраза
- не писать “смотри значение в переводе фразы”

---

## 30. Живой язык

“Живой язык” учит реальным фразам.

Формат:

```text
💭 Живой язык

> Dat is de druppel!

Это последняя капля.

Ещё можно сказать:
“Моё терпение лопнуло”, “Это уже перебор”.

Пример:
Eerst was mijn trein te laat, toen morste ik koffie over mijn shirt en nu dit?
Dat is de druppel!
```

Правила:

- фраза в quote-блоке
- без академических объяснений
- живой перевод
- один пример
- минимум вариантов
- правило максимум в 1-2 строки
- «Живой язык» доступен только по ручной кнопке и не входит в плановые уведомления
- уведомление «Слова и фразы дня» решает только задачу повторения ранее изученного материала

---

## 31. Игра-детектив

Игра должна учить через интерес.

Правила:

- один эпизод - одна маленькая задача
- без длинного квеста
- без перегруза текстом
- язык соответствует активному языку обучения
- уровень соответствует уровню пользователя
- подсказка помогает, но не решает всё
- финал закрепляет фразу или слово

---

## 32. Словари и языки

Раздел отвечает за:

- активный язык обучения
- уровень нидерландского
- уровень английского
- пользовательский словарь
- добавленные слова
- знакомые слова

Правила:

- язык обучения не прячется в общих настройках
- лишние языки не показываются во вкладке обучения
- активный язык указан явно
- смена языка влияет на все обучающие функции
- при ручном добавлении слова бот пишет, в каком виде оно сохранено
- слово или фраза сохраняются в словарь сразу, без ожидания подтверждения кнопкой; вместо кнопки добавления показывается кнопка удаления, на случай если разбор ошибся
- термин в карточке словаря выделяется жирным курсивом и с большой буквы
- перевод в карточке словаря всегда показывается одной строкой через жирную стрелку `→`, а не отдельной подписанной строкой «Перевод: …»
- заголовок статуса карточки («✅ Добавлено»/«✅ Обновлено»/«📚 Найдено») — отдельной строкой, без термина на той же строке
- пример предложения в карточке словаря — через `→` (не тире) и оформлен курсивом целиком

---

## 33. Досуг

Досуг - не каталог. Это подбор под вкус пользователя.

Направления:

- кино
- книги
- музыка
- концерты
- события
- что сейчас в кино

Правила:

- рекомендации объясняют “почему тебе”
- не подбирать случайное
- учитывать любимое и сохранённое
- не повторять уже известное без причины
- показывать коротко
- давать быстрые кнопки действия
- настройки досуга держать внутри досуга
- LLM использовать только для персонального вкуса и объяснения

---

## 34. Кино

Кино помогает выбрать, что смотреть сегодня.

Разделы:

- подбор кино
- жанры
- настроение
- любимое
- сохранённое
- что сейчас в кино
- настройки кино

Жанры:

```text
Комедия
Ужасы
Фантастика
Триллер
Романтика
Драма
```

Настроения:

```text
лёгкое
подумать
грустное
романтика
страшное
неожиданная концовка
захватывающее
атмосферное
необычное
масштабное
медленное и красивое
без остановки
спокойный вечер
вдвоём
на один вечер
запутанное
```

Правила:

- не путать жанр и настроение
- показывать причину рекомендации
- если это сериал - показывать сезоны, серии и статус
- если есть дата следующей серии - показать
- учитывать страну пользователя для кинотеатров
- для Нидерландов использовать актуальные источники, если источник подключён

---

## 35. Книги

Книги подбираются по вкусу, настроению и уровню нагрузки.

Правила:

- не предлагать случайные топы
- объяснять, почему подходит
- учитывать любимые жанры
- различать лёгкое и плотное чтение
- сохранять книги в контекстном разделе
- управление сохранённым делать внутри книг

---

## 36. Музыка и концерты

Музыка делится на две задачи:

1. Подбор музыки
2. Концерты

Не смешивать.

Подбор музыки:

- похожие исполнители
- настроение
- новые идеи
- почему подходит

Концерты:

- ближайшие события
- дата
- город
- площадка
- исполнитель
- связь с любимыми артистами

Правила:

- не показывать дубли концертов без причины
- сортировать события по дате
- не перегружать описанием
- если данных нет - сказать коротко
- если событие далеко - показать город явно

---

## 37. Поездки

Поездки помогают выбрать короткую поездку или маршрут.

Правила:

- учитывать город пользователя
- учитывать погоду и сезон
- давать реалистичный маршрут
- не предлагать сложный план без запроса
- писать время, транспорт и нагрузку
- для локальных мест проверять актуальность, если данные могут измениться

Формат:

```text
🚲 Идея на сегодня

Алкмар → Берген

Почему стоит:
> Недалеко, красиво и можно уложиться в полдня.

Как ехать:
Велосипед · около 40 минут в одну сторону
```

---

## 38. Готовка

Готовка помогает решить, что приготовить.

Разделы:

- завтрак
- обед
- ужин
- из того, что есть
- сохранённые рецепты
- настройки

Правила:

- не повторять рецепты подряд
- учитывать продукты пользователя
- разделять завтрак, обед и ужин
- показывать время и сложность
- писать конкретные шаги
- не делать длинные кулинарные статьи
- фото рецептов сейчас не используются (продуктовое решение) - не добавлять `photo_query_en`/`name_en`/`photo_fallback_queries` в промпт и не тратить AI/vision-запросы на фото

---

## 39. Здоровье

Здоровье - дневник и мягкие подсказки, не врач.

Разделы:

- самочувствие
- история самочувствия
- короткие check-in
- лагом
- простые действия

Правила:

- не ставить диагноз
- не делать медицинские выводы без оснований
- если есть риск - советовать врача или ветеринара
- для ADHD давать действие на 2-5 минут
- не писать философию
- не давить мотивацией
- история должна быть короткой и понятной

Лагом-тон:

```text
Сейчас не вся жизнь. Сейчас один шаг.
```

---

## 40. Сохранённое

“Сохранённое” показывает накопленные объекты пользователя.

Правила:

- не превращать в мусорную корзину
- группировать по разделам
- управление объектом делать в контексте
- удаление через confirm
- после удаления обновлять список
- не дублировать функции разделов

---

## 41. Настройки

Настройки нужны для общих параметров.

Там живут:

- профиль
- город
- уведомления
- общие предпочтения
- доступные параметры

Не переносить сюда управление объектами:

- любимые фильмы
- вещи гардероба
- слова
- рецепты
- книги
- музыка

Это должно жить внутри своих разделов.

---

## 42. Уведомления

Уведомления должны быть полезными и не шумными.

Правила:

- типы уведомлений брать из `settings.NOTIF_TYPES`
- не хардкодить список в админке
- пользовательские и админские списки опираются на один источник
- расписание сортируется по времени
- в админском тесте показывать время и тип
- не дублировать уведомления
- не слать старые сообщения после рестарта
- плановые уведомления не должны бесконтрольно вызывать LLM

---

## 43. Ассистент

Ассистент принимает свободные команды пользователя.

Примеры:

```text
добавь в словарь слово ...
добавь вещь в гардероб ...
сохрани фильм ...
напомни ...
что приготовить из ...
```

Правила:

- если действие распознано - выполнить или коротко подтвердить
- после добавления написать, что сохранено
- не отправлять пользователя искать раздел вручную, если можно сделать сразу
- если команда непонятна - задать один короткий вопрос
- не делать длинную лекцию

---

## 44. Админка

Админка должна быть компактной.

Правила:

- меньше кнопок верхнего уровня
- диагностика внутри “Система”
- API, LLM и новости можно объединять в одну системную диагностику
- тесты уведомлений держать во вкладке “Уведомления”
- кнопка “Проверить всё” не дублируется в разных местах
- все кнопки возврата только `⬅️ Назад`
- ошибки показывать одной строкой
- технические подробности прятать глубже
- лимиты API показывать просто, не сырой статистикой

---

## 45. Проверка API

Экран проверки API должен быть коротким.

Показывать:

- сервис
- статус
- использовано / лимит, если лимит известен
- осталось, если важно
- последнюю ошибку, если есть

Не показывать на главном экране:

- кэш
- повторы
- длинные raw errors
- stack traces
- лишние нули

Подробности можно держать во вложенном экране.

---

## 46. Новости

Новости должны быть персональными, свежими и локальными.

Важные темы:

- Нидерланды
- Алкмар
- Noord-Holland
- DUO
- NS
- gemeente
- миграция
- медицина
- AI
- OpenAI
- Apple
- кино
- музыка
- Telegram
- Cloudflare
- OpenWeather
- Pexels
- готовка
- досуг
- путешествия

Правила:

- для Нидерландов искать на нидерландском и английском
- локальные новости искать по Alkmaar и Noord-Holland
- не показывать старые новости как новые
- не выдавать одну категорию каждый раз
- объяснять “почему тебе”
- лучше 3 хорошие новости, чем 10 случайных
- если новость не касается пользователя - не показывать

Схема:

1. Без LLM собрать источники, убрать дубли, отфильтровать старьё.
2. LLM использовать только для выбора 3-5 важных новостей и объяснения “почему тебе”.
3. Результат кэшировать.

---

## 47. Онбординг

Онбординг собирает минимум данных, чтобы бот стал полезным.

Важно собрать:

- город
- язык обучения
- уровни языков
- уведомления
- базовые интересы
- стиль гардероба, если пользователь идёт в гардероб
- досуговые предпочтения, если пользователь идёт в досуг

Правила:

- не задавать лишние вопросы заранее
- лучше спросить в момент входа в раздел
- сохранять ответы в профиль
- не повторять вопрос, если ответ уже есть
- давать возможность изменить позже

---

## 48. Данные пользователя

Данные пользователя - память бота.

Правила:

- не терять существующие данные
- не перезаписывать профиль целиком без причины
- миграции делать осторожно
- новые поля добавлять с fallback
- старые профили должны продолжать работать
- если данных нет - показывать нормальный пустой экран, а не ошибку

---

## 49. Внешние API

Правила:

- не делать сетевой запрос, если есть свежий кэш
- не делать несколько запросов там, где достаточно одного
- учитывать дневные лимиты
- 429 не маскировать как обычную недоступность без причины
- fallback должен быть простым и честным
- внешние ошибки не показывать пользователю как raw error

---

## 50. Когда менять docs

Меняешь код - проверь docs.

Примеры:

- изменил гардероб - `docs/wardrobe.md`
- изменил обучение - нужный файл обучения
- изменил меню - `docs/menu.md`
- изменил админку - `docs/admin.md`
- изменил уведомления - `docs/settings.md` и `docs/instructions.md`
- изменил расписание - `docs/instructions.md`
- изменил удаление - `docs/cleanup.md`
- изменил deploy-уведомление - `RELEASE_NOTES.md` и при необходимости `docs/admin.md`

## 51. Рабочий алгоритм

Перед правкой:

1. Прочитай `CLAUDE.md`.
2. Найди связанный модуль.
3. Найди связанный `ui/*.py`.
4. Найди docs-файл раздела.
5. Проверь `RELEASE_NOTES.md`.
6. Посмотри стиль соседнего кода.

Во время правки:

1. Меняй минимально нужный участок.
2. Не ломай callback-и.
3. Не меняй UX соседних разделов без запроса.
4. Не добавляй тесты.
5. Не добавляй архитектуру без причины.
6. Следи за кнопкой `⬅️ Назад`.
7. Не добавляй прямой LLM-вызов в раздел, если есть AI-слой.

После правки:

1. Обнови docs.
2. Обнови `RELEASE_NOTES.md`.
3. Проверь сценарий вручную или через маленький `python3 -c`.
4. Кратко напиши, что изменено.
5. Честно напиши, что проверено.
6. Не обещай то, что не проверял.

## 52. Как отвечать владельцу проекта

Ответ после работы короткий.

Формат:

```text
Готово.

Изменено:
- ...

Обновлено:
- docs/...
- RELEASE_NOTES.md

Проверено:
- ...
```

Не писать длинный отчёт с пересказом каждого файла.

Не писать “комплексное тестирование”, если его не было.

## 53. Повторяющиеся правила

Если владелец второй раз объясняет одно и то же правило - это правило проекта.

Тогда нужно:

- внести его в `CLAUDE.md`, если оно глобальное
- внести его в `docs/*.md`, если оно про раздел
- внести его в `.claude/rules/`, если это правило кодинга
- не заставлять владельца повторять это снова
